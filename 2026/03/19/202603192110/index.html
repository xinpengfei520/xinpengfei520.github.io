Codex 调教指南：从安装到变现，手把手教你用 AI 写出能赚钱的代码

发表于 2026-03-19 更新于 2026-04-10 分类于 阅读 阅读次数： 本文字数： 4.4k 阅读时长 ≈ 4 分钟

你装好了 Codex，兴冲冲地输入第一句话：”帮我做一个能赚钱的 App。”

Codex 回了你一大堆代码。你看不懂。跑不起来。然后你关掉窗口，得出结论：”这玩意儿不好用。”

这个场景，每天在无数人身上重演。

问题不在 Codex。问题在于——你不会调教它。

Codex 就像一个超级厉害的实习生。能力极强，但需要你把需求说清楚。你说”做个 App”，它不知道做什么 App、给谁用、解决什么问题。你说”给用户列表加分页，每页 20 条，底部显示页码，点击页码切换”，它 3 分钟就给你做好了。

差距不在工具，在用法。

今天这篇文章，我把从安装 Codex 到做出第一个能赚钱的产品的完整流程，拆成 5 个步骤，逐一讲透。不需要编程基础。只需要一个想法，和一点耐心。

第一步：安装——3 种入口，选适合你的

Codex 有三种使用方式，对应三种人群：

方式 1：桌面 App（推荐新手）

最简单。下载、登录、开始用。

  • macOS：官网下载 .dmg 安装
  • Windows：2026 年 3 月 4 日已发布 Windows 版
  • 登录你的 ChatGPT 账号即可

桌面 App 的优势：有可视化的代码审查面板、支持多线程并行、支持 Git 工作树。你可以同时开 5 个任务，每个任务在独立的沙箱里运行，互不干扰。

免费用户也能用，只是额度有限。Plus 用户（$20/月）额度翻倍。

方式 2：命令行 CLI（推荐有终端经验的人）

首次运行会弹出浏览器让你登录 ChatGPT 账号。CLI 的优势是速度快、可脚本化、支持 MCP 协议连接外部工具。

方式 3：Web 版（最轻量）

直接在 ChatGPT 里用。打开 chatgpt.com/codex，选择一个仓库，开始提任务。任务在云端沙箱运行，不占用你本地资源。

我的建议： 如果你是第一次用，直接下载桌面 App。门槛最低，体验最好。

第二步：前 3 个任务——别急着造产品，先学会”说话”

很多人装好 Codex 的第一件事就是提一个巨大的需求。然后翻车。

正确的姿势是：前 3 个任务，只做小事。

任务 1：让它解释代码

找一个你感兴趣的开源项目，打开一个文件，问：

“解释一下这个文件的作用，每个函数做了什么？”

这一步的目的不是写代码，是观察 Codex 怎么工作。它会怎么组织答案？详细到什么程度？你对它的输出风格有了基本感知。

任务 2：做一个极小的修改

“把这个按钮的颜色从蓝色改成绿色，然后运行测试确认没有破坏其他东西。”

注意这句话的结构：做什么 + 验证条件。这是 Codex 最喜欢的 prompt 格式。

任务 3：做一个带验证的完整小任务

“给这个表单加一个邮箱格式验证。如果格式不对，在输入框下方显示红色错误提示。完成后运行 lint 和单元测试。”

三个任务做完，你已经掌握了 Codex 的核心工作循环：读 → 计划 → 编辑 → 测试 → 审查。

第三步：调教——让 Codex 真正懂你

“调教”这个词用在 Codex 上，核心是三件事：AGENTS.md、Skills、和 Prompt 技巧。

3.1 AGENTS.md——给 Codex 一本员工手册

AGENTS.md 是一个 Markdown 文件，放在你的项目根目录下。Codex 每次启动都会读它，就像新员工入职第一天读公司手册。

你应该在里面写什么？

有了这个文件，你每次提需求都不用重复解释技术栈和规范。说一次，永远生效。

AGENTS.md 支持分层——你还可以在子目录放更具体的指令。比如 src/components/AGENTS.md 专门指导组件开发规范。

3.2 Skills——把重复的事变成一键操作

如果你发现自己反复在写同样类型的 prompt，就该把它变成一个 Skill。

比如你经常需要”给新页面创建路由 + 组件 + 测试文件”，可以创建一个 Skill：

下次你只需要说：”用创建新页面的 Skill，创建一个 Dashboard 页面。”

Codex 桌面 App 内置了一些官方 Skills，比如从 Figma 设计稿生成前端代码、自动生成 PR 审查报告等。

3.3 Prompt 技巧——5 条黄金法则

法则 1：先让它出计划，再让它动手。

不要说”帮我实现这个功能”。要说：

“先展示你的实现计划，分 3 步说明。确认后再开始写代码。”

这一招能减少 50% 以上的返工。

法则 2：定义”完成”的标准。

不好的 prompt：”加个搜索功能。”

好的 prompt：”加一个搜索框，支持按标题模糊搜索。搜索结果实时显示在列表中。完成标准：搜索功能正常、现有测试通过、lint 无报错。”

法则 3：限定范围和禁区。

“只修改 src/components/Header.tsx 文件。不要动 App.tsx。不要改变现有的 props 接口。”

法则 4：给参考示例。

如果你想要特定的风格，给它看一个例子比文字描述有效 100 倍。你可以直接截图发给它——Codex 支持图片输入。

“参考这张截图的布局风格，帮我做一个类似的定价页面。”

法则 5：一次只做一件事。

不要在一个 prompt 里塞 5 个需求。拆开，每次一个。做完一个，确认没问题，再做下一个。

第四步：做产品——从想法到上线的完整路径

调教好了 Codex，现在开始做正事。

第 1 阶段：验证想法（30 分钟）

不要上来就做完整产品。先做一个最小可用原型。

你有一个想法：”做一个帮自媒体人批量生成多平台文案的工具。”

告诉 Codex：

“帮我做一个单页 Web 应用。功能：用户输入一段文字，点击按钮后，分别生成适合微博（140 字内）、小红书（带 emoji 和话题标签）、LinkedIn（专业商务风格）的版本。纯前端实现，用 HTML + CSS + JavaScript，不需要后端。”

30 分钟内，你就有了一个能演示的原型。发给朋友试用，收集反馈。

第 2 阶段：迭代打磨（2-3 天）

原型验证后，开始加功能。记住：每次只加一个功能，用对话式开发推进。

  • “加一个历史记录功能，保存用户之前生成的文案”
  • “支持自定义 AI 模型——让用户选择用 GPT 还是 Claude”
  • “加一个设置页面，让用户保存自己常用的品牌调性描述”
  • “做成 PWA，让用户可以安装到手机桌面”

每加一个功能，都让 Codex 跑一遍测试。

第 3 阶段：上线（1 天）

Codex 不只能写代码，还能帮你部署：

“帮我把这个项目部署到 Vercel。生成 vercel.json 配置文件，确认构建命令正确。”

再做一个落地页：

“帮我做一个产品落地页，介绍这个工具的三个核心功能，包含截图展示区、定价表（免费版 vs 付费版）、FAQ、和注册表单。”

域名买一个（$10/年），绑定到 Vercel。上线。

第 4 阶段：用 Automations 自动化运营

Codex 的 Automations 功能让你可以设置定时任务——写好 prompt，设定频率，Codex 自动在后台运行。

实际用法：

  • 每天早上自动检查用户反馈：扫描邮箱和 GitHub Issue，整理成待办清单
  • 每周自动生成 changelog：汇总本周代码提交，生成更新日志
  • 每次 PR 自动审查：有新 PR 就自动审查代码质量和安全性

这不是”以后可能用到”的功能——这是让你一个人当一个团队的核心杠杆。

第五步：变现——4 种赚钱路径

产品做好了，怎么赚钱？

路径 1：SaaS 订阅

最经典的模式。免费版提供基础功能，付费版（$5-20/月）解锁高级功能。适合工具类产品。

PSPDFKit 的创始人 Peter Steinberger 用 Codex 后说，他的开发效率提升了大约一倍。他自称是”Codex 最大的免费推广员”——后来收到了 OpenAI 和 Meta 的收购邀约。

路径 2：Chrome 扩展 / 浏览器插件

开发成本低，分发渠道现成（Chrome Web Store），可以用订阅制或一次性付费。前面几篇文章提到的那位产品经理，用 Codex 做了一个 Notion 辅助插件，三周收入 $400+。

路径 3：模板 / 组件市场

用 Codex 批量生成高质量的 UI 组件、落地页模板、邮件模板，放到 Gumroad、Lemonsqueezy 等平台售卖。一个模板卖 $19-49，做 10 个就有了一个小产品线。

路径 4：接外包，用 Codex 10 倍提效

传统外包开发一个网站要 2-4 周。你用 Codex 可能 2-3 天搞定。客户付的是市场价，你花的是 1/10 的时间。利润率暴涨。

Root Ventures 的投资人 Lee Edwards 说得好：对于能充分利用 AI 工具的人来说，**”一个人就是一座软件工厂”**。

3 个容易踩的坑

最后说三个新手最容易犯的错误：

坑 1：prompt 写得太模糊。 “帮我做个好看的网站”——什么叫好看？给谁看？什么风格？越具体，结果越好。

坑 2：不审查就接受代码。 Codex 生成的代码大部分时候是对的，但不是每次都对。尤其是安全相关的代码（用户认证、支付逻辑），必须人工审查。永远不要 auto-accept。

坑 3：一开始就做大项目。 不要第一个项目就想做下一个 Notion。从一个解决你自己痛点的小工具开始。做小、做快、做完、上线。然后在用户反馈中迭代。

写在最后

Codex 不是魔法棒，不能帮你凭空想出一个好主意。

但如果你已经有了一个想法——一个痛点、一个需求、一个”要是有这个工具就好了”的念头——Codex 可以帮你把它变成现实。

以前，想法和产品之间隔着一道叫”编程”的墙。现在这道墙正在消失。

你的想法值多少钱，不再取决于你会不会写代码。取决于这个想法本身够不够好，以及你愿不愿意花三天时间把它做出来。

打开 Codex，今晚就开始。
