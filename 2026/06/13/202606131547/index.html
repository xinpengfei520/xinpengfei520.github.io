我把整套家当押在 Claude 上，直到看见它被政府一键叫停

发表于 2026-06-13 更新于 2026-07-01 分类于 观点 阅读次数： 本文字数： 2.4k 阅读时长 ≈ 2 分钟
6 月 12 日下午，美政府以国安为由叫停 Anthropic 最强的 Fable 5 和 Mythos 5，当天对所有客户禁用。我的写作流水线挂着十几个 skill，全长在 Claude Code 上，这事让我看清：把生产力押在单一模型供应商身上，它说没就没，控制权不在你手里。该认真养个国产后端做 plan B，不为省钱，为主力没了那天还能照常开工。

哈喽，我是飞飞。

6 月 12 日下午 5 点 21 分，Anthropic 收到美国政府一纸指令，把自家最强的两个模型 Fable 5 和 Mythos 5，当天就对所有客户禁掉了。

我刷到这条的第一反应，谈不上吃瓜。后背先凉了一下。

凉的不是这件事本身有多大，是它戳破了一个我一直假装没看见的事实：我每天干活最依赖的那个能力，开关根本不在我手里。它说没，就能没。

这事得从政府给的那个理由说起。因为那个理由，离我天天干的活近得有点吓人。

让模型下架的那个「越狱」，就是我每天让它干的活

政府叫停的官方说法，是国家安全。他们说，发现了一种能「越狱」Fable 5 的方法。

这个所谓的越狱具体是什么呢？是让模型读一段代码，找出里面的软件漏洞，再帮你把它修掉。

你要是写代码的，看到这儿应该跟我一个表情。这不就是 Claude Code 每天帮我干的事吗？读仓库、挑 bug、给补丁，我一天得让它干几十遍。

Anthropic 自己也不服。它公开回了一句，大意是：仅凭一个狭窄的、还没坐实的越狱可能，就召回一个已经给数亿人在用的商业模型，不合理。

它还补了一刀，说这个能力 OpenAI 的 GPT-5.5 也有，安全圈的人天天拿它做防御，凭什么单挑我家。

更微妙的是，政府拿出的证据只是口头的。Anthropic 自己审完觉得，这技术也就能挑出几个早就公开的小毛病。

但吵归吵，模型当天就是没了。理由站不站得住是一码事，它说停就停，是另一码事。后面这半句，才是真正让我那天晚上睡不踏实的地方。

我到底把多少身家，押在了一家我管不着的公司身上

那天晚上我做了件事，把自己的家底认真盘了一遍。

光是写公众号这一条流水线，我就挂了十几个 skill：拉资讯的、查重的、配图的、发草稿箱的。外加一份越喂越厚的 CLAUDE.md，一摞 memory。这些东西全长在 Claude Code 认 skill、读 frontmatter、调 hook 的那套脾气上。日常写代码也在上面跑。说白了，这是我每天开工的地基。

还有一层账，平时我不太挂在嘴上。每个月给 Anthropic 的钱不算少，所以让 agent 干一件正经活之前，我都会下意识掂量一下，这活值不值得动用最贵的 Opus，杂活就手动切个便宜模型省点额度。这种用之前先在心里过一遍秤的绷着感，本身就说明我把它焊得有多死。

我没打算回头骂谁。Claude 好用是真好用，这套地基也是我自己一砖一瓦垒上去的，自愿押的，没人逼我。

可这次的事，把一个我平时绝口不提的风险，直接摆到了台面上。

这回的指令，针对的是 Fable 5 和 Mythos 5 对外国人的访问。我日常 Claude Code 调的是别的模型，暂时没断，这次轮不到我慌。

但它完整演示了一套流程：一个外部指令下来，一家公司可以在几个小时内，让一批人手里的工具集体失灵。今天的理由是国安，掐的是这两个模型。哪天换个理由、换个型号，我那条流水线，完全可能某天一早醒来就跑不动了。

单点依赖这东西，平时一点感觉都没有，顺手得很。出事那天你才发现，自己整个生产力，全系在一根别人攥着的绳上。

那我有没有 plan B，我上周真去试了一把国产后端

这个问题，我没让它停在脑子里空想。上周我就真装真跑过一套国产备胎。

那东西叫 MiMo Code，小米的终端编程助手。macOS 上一行 npm 命令，十秒装好。我走的是它内置的免费通道，1M 上下文，整套任务跑完，账单是 0。最让我意外的是，它安装的时候，把我那 44 个 Claude Code 配置一键导了进去，同步了 5 个。备胎主动上门认亲，这点确实有诚意。

当然也踩了坑。我用非交互模式直接喂任务，它挂死了 14 分钟，零输出，连个错都不报。后来才搞明白，免费通道得先手动跑一次「铸 token」。把令牌铸出来，再冒烟，11 秒就响应了。

我让它写一个零依赖的小 CLI，顺手让它自己配上测试。第一轮 5 个用例，3 过 2 败。那 2 个挂掉的，是列表解析出了岔子：它写的那条正则，把 tags 那一行的空值抢先匹配掉、赋成了 null，底下的列表项没人接管，本该是数组的字段，落成了一个孤零零的 null。

但接下来那一段，才是我决定认真对待它的原因：它自己写的测试，抓出了它自己的 bug；它读着报错自我诊断，那段话跟我手动翻代码得出的结论一字不差；然后一次 Edit 改对，重跑 5 个全过。

它官网挂的跑分其实挺好看，SWE-Bench Pro 上比我在用的高出几个点。但我没太把那张表当真，那是它自己跑的、还没上官方榜，口径跟正式提交对不齐。真正让我高看它一眼的，是它栽进 bug 又自己爬出来的那一下。

实话讲，跑通一个小任务，不等于能无缝平替我的主力。我那套专门防 AI 味的 grep 规则、十几个 skill 吐工具格式的脾气，换个模型底座，十有八九都会飘。挨个重新校准，够我搭进去好几个晚上。这笔迁移的时间账，省模型钱那本账上从来不记。

所以 plan B 从来不是按一个开关就能切过去的东西。它得你另起一套，平时就养着。但养着一个能跑起来的备胎，和手里一个都没有，真出事那天的踏实程度，完全是两回事。

别把你的「想」和「做」，全外包给同一个 AI

把这事往大里想一层。这两年，我们其实在飞快地把越来越多的「想」和「做」，外包给某一个 AI。

越外包越顺手，越顺手越离不开。可你越离不开，那家公司对你的议价权就越大，你自己的退路就越窄。

说到底，这次是政府一纸指令，下次换的可能是涨价，可能是某个你天天用的功能被悄悄砍掉，也可能是它哪天自己不干了。开关的样式会一直变，开关不在你手上这件事，不会变。

我不会因为这一下就弃用 Claude，它还是我的主力，我也不打算装什么清醒。但我从这周起，开始认真养第二套后端。不为省那点模型钱，为的是哪天主力真没了，我还能照常开工，不至于干瞪眼。

国产后端到底能不能扛起我整条流水线，我还在试，坑也还不少，远没到能拍胸脯的程度。等我把它跑顺了，单独写一篇摆给你看。

最后想问问你：你手里那个天天用、已经离不开的 AI 工具，要是明天早上突然用不了了，你能在多长时间内，换上别的、不耽误手里的活？
