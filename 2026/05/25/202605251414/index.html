Google 一周两次 3x，把 Antigravity 周配额拉到 9x 之后我重新算了笔账

发表于 2026-05-25 更新于 2026-07-14 分类于 观点 阅读次数： 本文字数： 5.7k 阅读时长 ≈ 5 分钟
Google在I/O后5天内两次3x、累计9x提升Antigravity周配额，起因是5月19日切换compute-based计费后大批Pro用户几小时烧光周额度，文章反推新算法由请求复杂度、功能用量与上下文长度三个维度构成，Pro与Ultra档改为5小时滚动刷新机制。

哈喽，我是飞飞。

上周三凌晨我在 Reddit 上刷到一个帖子，标题写着「weekly quota burned in 1 hour」。点进去发现是个 Antigravity Pro 订阅用户，账号刚续费完一周，开了一个 Composer session 跑重构，吃完午饭回来配额提示框就弹出来了。

第一反应是这哥们儿是不是误操作。

然后下面跟的是几十条同款吐槽。「I/O 之后我的 weekly quota 撑了 40 分钟」「Pro 订阅 22 美刀一个月，跑两个 multi-agent 任务就到顶了」「上周还能跑一整天，这周开两个 agent manager 同时跑就锁」。

我去 X 上搜 Logan Kilpatrick 和 Varun Mohan 的最新 tweet，刚好赶上 Google 这一周内两次 3x 提升 Antigravity 内的 Gemini 配额。Logan 的原话是 「We just 3xed the Antigravity limits again, but this time, the weekly quotas. We want you to stay in flow state building with Gemini」。

5 天内两次 3x，累积 9x。

但你打开 Reddit 评论区往下翻第二条高赞 reply，写的是「even 9x is still less than what we had before May 19」。

这件事不是单纯配额涨了 9 倍。

这一篇我把 Reddit 大批用户烧光周配额的 timeline、Google 5 天 9x 的真实算法逻辑、跟 Cursor / Claude Max 同档位的对比，还有国内 Pro 订阅用户接下来怎么决策，按「看到的现象 → 真实机制 → 国内影响」梳一遍。

I/O 之后的两天，Reddit 上集体炸了

5/19 周二 keynote 直播上 Sundar 宣布 Antigravity 2.0 拆 5 surface 那一刻，全场 IDE 工程师在弹幕里欢呼。当天晚上社区氛围特别好，大家都在装 Antigravity 2.0 desktop app 准备试 Agent Manager。

到周三凌晨 8 点开始，r/google_antigravity 这个版块连续刷出「quota burned」帖。

写代码的人都知道 agentic IDE 是个什么场景。

你打开 Antigravity 2.0，丢一个目标（比如「帮我重构这个 React 组件的 props 管理」），Agent Manager 自动开 3-4 个子 agent 并行跑。一个改 hooks、一个改类型、一个跑测试、一个查文档。每个子 agent 都是完整的 Gemini 3 Pro context、都在做 tool calling、都在写文件。

一次 work session 跑 30 分钟，消耗的 token 量可能等同于 ChatGPT 用一整天。

但你的 dashboard 是按「周」计的。

5/19 之前的旧规则下，Pro 订阅一周大概能跑 100 个完整 Composer task。5/19 当天 Google 静悄悄换了一套 compute-based 计费，相同的 100 个 task 在新算法下被判为消耗超额，账户提示 「weekly limit reached」。

工程师不会忍这种事。

Reddit 帖子 24 小时之内热度炸到论坛主页置顶，X 上 #antigravity 标签开始挂 quota 截图。Varun Mohan 周三下午就出来发声了。

Logan 那条 X 公告把 work session 这件事重新定义了一遍

Varun Mohan 周三晚上发的第一条 X 是这样：「If you are on a paid plan, your Gemini quota has been reset for the week and increased by 3x moving forward. Keep building!」

这个解决方案乍一看够诚意。

但 24 小时后 Reddit 上又开始第二轮吐槽：「3x 之后我又烧光了」「Pro 一档配额涨了，但跑 2 个并发 Composer 还是触顶」。

周四晚上 Logan Kilpatrick 在 X 上贴了第二次公告：「We just 3xed the Antigravity limits again, but this time, the weekly quotas」。同时 Varun Mohan 补充：「acknowledging that users could hit their weekly limits after a couple work sessions」。

这一句「couple work sessions」是 Google 这次最关键的一句话。

说白了，Google 内部以前的计费假设是「engineer 用 IDE 跟用 ChatGPT 一样，问一句答一句、间断使用」。但实际工程师把 Antigravity 当连续 8 小时高强度上下文窗口在用。一个 work session 就是 1-3 小时连续 agentic loop，里面有几十个子 agent 在跑，每个子 agent 都是完整 reasoning 链。

5 天连刷两次 3x 这个动作，等于 Google 在 48 小时内承认自己计费 anchor 设错了。

Google product team 在工程师群体集体反弹下，48 小时内把「engineer 怎么用 IDE」这个底层假设重新校准了一遍。这种响应速度在大厂里少见。

compute-based 这套算法在算什么

Google 这次新计费里 「compute-based」 这个词出现了无数次，但官方 blog 写得很模糊。我把 9to5google / AndroidHeadlines / Google blog feed 这几篇连起来读，反推出真实的算法构成。

旧算法（5/19 之前）：按 daily prompt 数算，简单粗暴。

新算法（5/19 之后）：compute-based，由 3 个维度共同决定。

请求复杂度：同样是问一句话，跑一次简单 chat completion 和触发一次 multi-step tool calling，消耗差几个数量级。Antigravity 2.0 里如果你的 prompt 触发 Agent Manager 并行开 4 个 sub-agent，这一次请求的 cost 等于以前 40 次普通 chat。

用到的功能：Browser integration 跑一次网页抓取加 DOM 解析、CLI 工具跑一次 bash command、File system 读写一次大目录，每个 surface 的 cost weight 都不一样。

聊天历史长度：每次请求都把整个对话 context 重传一遍，session 越长、context 越大，单次请求的 token cost 越高。

Cursor 还在按 request 数算（Pro 一档每月 500 fast premium request）。Claude Code 按 token 算（input 和 output 两条独立曲线）。Google 这次直接对齐「代理实际产生的计算」，把 prompt 复杂度、surface 用量、context length 全揉进去。

本质上这是 agentic IDE 时代必然的范式转移。

按 prompt 数算配额，是 chat 时代留下来的逻辑。agentic IDE 一次任务消耗的 compute 是 chat 一句话的几十倍上百倍，按 prompt 数算等于让 multi-agent loop 用户用很便宜的价格占用很贵的资源，平台经济学不成立。

compute-based 是对的方向。Google 这次只是切换时间点没选好（I/O 之后 5 小时内）加上起始 baseline 给得太低（被工程师当天就烧光），后面 5 天内两次 3x 是把 baseline 拉回合理位置。

Pro 和 Ultra 拿到的 5 小时滚动刷新，是 Anthropic Claude Max 同款机制

Google 官方 blog 这次明确写：「Pro / Ultra subscribers now receive priority access, featuring our highest, most generous rate limits with quotas that refresh every five hours」。

每 5 小时刷新一次的机制不是 Google 原创。

Anthropic Claude Max（200 美元 / 月那个档）就是 5 小时滚动窗口刷新。Claude Max 用户每 5 小时拿一个 quota 重置，理论上一天能撑 4-5 轮高强度使用。Cursor Pro 还是按月计的 fast premium request 数，没有滚动窗口概念。

Google 这次在 Antigravity 上把 Pro / Ultra 改成 5h 滚动刷新，简单讲是抄 Claude Max 的玩法。

这套机制对 agentic IDE 用户更友好的原因是：work session 是 1-3 小时高强度集中型，不是均匀分布。让你每 5 小时拿一次 quota 重置，比「每周给你一个大额度自己分配」对重度工程师更合理。

但 Google 这次抄的是机制，不是数字。Claude Max 一个月 200 美元，Antigravity Pro 一个月 22 美元。同样的 5h 刷新机制下，22 美元档的 baseline 配额能拉到什么水平？9x 之后大概拉到 Cursor Pro 的同价位（20 美元）不相上下、比 Claude Max 同价位（不存在，Max 没有 20 档）便宜得多。

但前提是 baseline 不再缩水。

国内开发者真正受影响的，就两类人

国内 Pro 订阅 Antigravity 的用户我大概观察了几个圈子，真正会被 9x 这事影响决策的就两类。

重度 multi-agent 工程师：日常在 Antigravity 里跑 Composer 任务、连续 4-6 小时高强度编程。这类用户 Pro 一档 22 美元 / 月在 9x 之前撑不过 3 天，9x 之后大概能撑到一周末。但如果你的 work session 是「打开 Antigravity → Agent Manager 同时开 4 个子 agent 重构整个 microservice → 跑 2 小时」这种规模，Pro 还是不够，得上 Ultra（250 美元 / 月）。

轻度单文件 refactor 用户：日常只用 Antigravity 写 React 组件、改 SQL、跑单文件 refactor。这类用户其实没怎么感受到 5/19 那次紧缩，9x 之后更是完全无压力。Pro 一档对你来说 quota 是冗余的。

剩下中间那一档「想用但不知道值不值」的人，实话讲这次 9x 的信号是积极的，可以考虑订一档。但要注意：现在的 9x 是相对 5/19 那个被骂回去的 baseline 的 9x，距离 5/19 之前的旧水平还有 30-40% 的差距。这意味着 Google 后面如果工程师群体不再激烈反弹，可能不会再继续 3x。

国内开发者的支付通路依旧是个坎：Wise、OneCard、美国 Apple ID 加美国信用卡。这条路径不变。Antigravity 也没有像 Cursor 一样开通过 GitHub Marketplace 等折中支付方式。

Free tier 被甩进 weekly cap，是 paid-first 模式定型的信号

5/19 之前 Antigravity 还有相对友好的 free tier 给开发者尝鲜。5/19 之后 Google 官方 blog 写：「For users on the free plan, we’ve shifted to a larger, weekly based rate limit to minimize hitting rate limits quickly during a project」。

中文翻一下：免费用户不再有 5h 滚动刷新，改成 weekly cap，每周给你一个大额度自己分配。

这个变化对偶尔试 Antigravity 一下的开发者影响不大。但对希望「白嫖 Antigravity 跑一些日常开发」的人是个明显劝退信号。Google 在用产品分层把 Antigravity 推向 paid-first 工具，免费用户能用但不能日常用。

跟 Cursor 走的同一条路。Cursor 在 2024 年底也做过类似切换，free 一档从「日常可用」压到「试用为主」，Pro 一档 20 美元变成日常用户的入门门槛。

对 Google 来说这是个商业模型修正。Antigravity 2.0 的 IDE / CLI / SDK / Web / Agent Manager 5 个 surface 每个都烧 compute，没有 paid-first 模式平台经济学撑不住。

但对国内开发者群体心理上的冲击不小：之前 Gemini 3 Pro 通过 AI Studio 拿到几乎是免费的，Antigravity 2.0 不能再走同套逻辑。要长期用，得付费。

9x 之后我要重新算的几笔账

发到 production 之前我自己接下来一周要做这几件事。

算 Pro 一档真实够不够：从今晚开始记录自己每次 Antigravity work session 的 compute 消耗（Antigravity 2.0 dashboard 里有详细 token usage 数据），跑一周下来看 Pro 22 美元那档真实能撑住几个连续 multi-agent session。稳稳够用就续 Pro，天花板还在频繁触发就上 Ultra。

对比 Cursor Pro 同价位：我手上同时有 Cursor Pro 订阅，下周打算做一个同任务对比。同一个 microservice 重构任务分别在 Antigravity Pro 9x 后和 Cursor Pro 500 fast premium 下各跑一遍，看真实 work session 完成数。

跑一次 Agent Manager 多 surface 并发实测：我之前一直只在 Antigravity IDE 里用，没有把 CLI、Web、IDE 同时跑过。9x 之后 baseline 够宽了，下周可以测一次极限：3 个 surface 并发加 Agent Manager 同时开 4 个子 agent，看真实压力下 compute 怎么走。

跑完会写一篇实测报告，包括 token usage 真实数据、各 surface compute weight 估算、9x 之后真实能稳定撑多久。

如果你也在订 Antigravity Pro 或 Ultra，9x 之后你的 work session 体感变化怎么样？周配额一周能撑几天？
