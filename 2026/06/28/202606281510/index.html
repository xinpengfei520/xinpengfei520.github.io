一个人开一家虚拟公司：把一队 agent 当部门经营，你坐的是 CEO 位

发表于 2026-06-28 更新于 2026-07-08 分类于 个人成长 阅读次数： 本文字数： 2.6k 阅读时长 ≈ 2 分钟
把一队 agent 当一家虚拟公司来经营，你坐的是对业务产出负总责的 CEO 位。ChatDev 让 agent 分演 CEO、程序员、测试搭虚拟软件公司，MetaGPT 把招团队和投预算写进代码、跑项目前要先给这家 AI 公司投一笔美元。我这批课就是当 CEO 派一队 agent 并行产的，管的是定选题、控模型成本、逐篇兜质量。它现在产出的是内容，还不是钱，这条我说在前头。

哈喽，我是飞飞。

我前阵子翻一个开源项目的代码，看到一行写法挺离谱。它不 new 一个对象，写的是 company.hire([ProductManager(), Architect(), Engineer()])，一口气把产品经理、架构师、工程师都招了进来。下一行 company.invest(3.0)，给这家公司投三美元的预算。再丢一句「做个贪吃蛇」，这帮 agent 就开始你来我往把活干完。我盯着这段代码看了半天，心里冒出一个念头，这哪是在调框架，这是在开公司。

这项目叫 MetaGPT，GitHub 上挂的简介就一句话，号称「第一家 AI 软件公司」。我后来又翻到清华那帮人做的 ChatDev，更直接，让一群 agent 分头扮 CEO、CTO、程序员、测试，开一场场专门的研讨，把一个软件从设计、写码、测试到出文档全跑完。看完我没急着去试，反倒愣了一下。因为这套「把 agent 编成一家公司」的玩法，我自己已经干了大半年，只是从没意识到，我坐的那个位置，叫 CEO。

当老板和当工头，到底差在哪一步

我之前在这套课里写过两篇 AI 团队的。一篇讲怎么给这支队伍配齐角色，研究的、写作的、SEO 的各就各位；另一篇讲怎么画清楚谁的活交给谁、信息在角色之间怎么流。那两件事解决的是同一个问题，团队怎么搭起来、怎么不散架。

但搭好之后还有一层，叫经营。工头管的是「这摊活做完没」，盯的是任务。CEO 管的是另外几件事，这家公司今天有没有产出拿得出手的东西、这摊生意值不值得做、钱烧得划不划算。你要盯的不是哪个 agent 更聪明，是这家公司整体跑出来的东西能不能见人。

位置一换，看问题的高度就变了。你不再趴在某个 agent 的输出上挑刺，而是往上抬一层，看整条业务线通不通。这就是从「用 AI」「管 AI」再往上走的那一步，把一队 agent 当一家公司来经营。

把这队 agent 排成部门来经营

我这家一人公司，部门是按业务线分的。产品部定选题，决定这期到底做什么；内容部把选题变成稿，调研、起草、润色一条龙；增长部管文章发出去之后怎么被人看见、怎么被 AI 引用；美术部出封面，给公司撑那张门面。

这跟 ChatDev、MetaGPT 给的那套角色是一个逻辑，都按公司职能去切。MetaGPT 内部就分产品经理、架构师、项目经理、工程师、QA，跟一家真软件公司的编制几乎一一对应。

说点我这家公司的实在家底。光写文章这一条线，背后就挂着十几个各管一摊的 skill，拉资讯的、查重的、配图的、发公众号的，再加一份越喂越厚的写作规则文件、一摞记着我脾气的 memory，整个长在 Claude Code 上。这堆东西就是我这家虚拟公司的固定资产，每个 skill 磨顺都搭进去过好几个晚上。

具体每个角色怎么配、交接口怎么对齐，我前两篇写过了，这里不重复。我只想点一句经营上的关键。当老板别陷在「这个 agent 干得好不好」这种事务里，部门对齐的是同一个业务目标，今天稳定产出一篇能发的文章，而不是各自把手里的任务做完就算交差。一个 agent 把活干得漂亮，但整条线没串起来，这家公司照样不出货。

一家虚拟公司，老板先要定做什么

CEO 手里第一份活不是派活。先得定这家公司做什么、不做什么。

放到我这儿，选题就是最核心的产品决策。我每天能调动的 agent 算力、我自己能过手的精力，都是有限的，这等于公司的产能就那么大。所以哪个选题上、哪个毙掉、这一系列课按什么顺序排，这根线我攥得死死的，不撒手。

这块我外包不出去。agent 能帮我把一堆候选选题打分、列出哪个好写哪个难写，但「这家号这个月主打什么、要立什么调性」这个拍板，是老板的活。我也试过让 agent 自己挑题自己写，跑出来的东西工整是工整，就是没魂，因为它不知道我这家号到底想成为谁。方向感这东西，目前还塞不进任何一个 agent 的脑子里。

给公司投一笔预算，CEO 绕不开这本账

MetaGPT 那个 invest 的设计，我觉得特别诚实。它官方注释写得明明白白，作为投资人，你给这家 AI 公司投入一笔美元，跑项目前这笔预算就得先到位，花的是调模型的 API 费。社区里有人实测过，一个稍微复杂点的应用，几美元到几十美元就这么烧进去了。

这正好戳中我每天的体感。每个月给 Anthropic 的钱不算少，让一个 agent 去干正经活之前，我都会下意识掂量一下，这活值不值得动用 Opus，纯杂活就手动切到便宜模型省点额度。这种「用之前先在心里过一遍秤」的绷着感，就是当老板要算的那本账。

工头不用管钱，CEO 必须管。一队 agent 并行开跑确实爽，效率肉眼可见地高，但 token 是真金白银，并行铺得越开，烧得越快。能不能把成本这本账一直拎在手里，是认真经营和图个新鲜瞎玩之间，最实在的那条分界线。

出了事没人会主动告诉你

经营里最坑的，是另一件事。

我的博客整条发布是全自动的，代码推上去就自动构建、自动部署。前阵子我踩了个血淋淋的坑，一个负责部署的 token 悄无声息过期了，部署连着静默失败了一整天。构建那一步明明是绿的，只有最后推送那一下红了，但平时没人盯着这个状态，我是隔了一天主动去查才发现的，镜像站挂了整整一天没人知道。

这就是一家公司最怕的场景，某个部门悄悄停摆，业务报表上还显示一切正常。agent 越自动、并行铺得越开，越容易冒出这种事，活看着是干完了，其实某一环早飘了，没人吭声。所以我这条流水线最后一定留着一道我亲自过手的闸，逐篇看稿、对外署名、出了岔子我自己担。这道闸我不交给任何一个 agent，因为对外扛事的是这家公司，是我这个人。

你可能以为当 CEO 是甩手享福，恰恰拧着来，越自动你越得在那几个关键节点上盯死。

这家公司现在产出的是内容，还不是钱

有句话我得说在前头。这套「一个人开一家虚拟公司」的玩法，确实能让我一个人稳定产出一批还过得去的内容，这是真的。但它没让我开成一家赚钱的公司。那些「一个人靠 AI 年入百万」的故事你听听就好，至少我没做到，我做到的是用它把产能拉满、把一堆手动环节砍掉，仅此而已。

绕回开头那行代码。那帮人把「开公司」三个字写进代码的时候，多半也就图个好玩。但你要真照着搭一套自己的虚拟公司跑顺，会发现最难的从来不是让 agent 扮好 CEO 这个角色。难的是你自己得真的像个 CEO 那样去想，这个月做什么、预算烧多少、出了事谁来兜底。角色是给 agent 演的，老板这门功课，得你自己补。

如果让你也开这么一家一个人的虚拟公司，你第一个想招进来的「部门」是哪个，是替你定方向的产品部，还是替你扛脏活的执行部？评论区聊聊。
