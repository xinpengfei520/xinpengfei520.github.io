浏览器住进了 Claude Code：agent 抓网页省掉一层，但我只敢先让它看

发表于 2026-07-10 更新于 2026-07-14 分类于 工具 阅读次数： 本文字数： 2.6k 阅读时长 ≈ 2 分钟
Claude Code 桌面版新增应用内浏览器，agent 干活时能直接开网页看内容，省掉你自己搭 Playwright 或 MCP 那层抓取工具。我那条用 playwright-fetch 抓网页喂写稿 agent 的老链路能省一截。但浏览器让 agent 的手伸到真实网页上，能点删除、能提交表单，撤不回的操作离它更近，所以我只敢先让它看、不敢让它动。

哈喽，我是飞飞。

先撇清一下。今天我已经连着写了 AI 味、Claude 反思、还有把 skill 当资产管那篇。这篇还是 Claude 生态的，但换了个具体的活：Claude Code 桌面版偷偷把一个浏览器塞进来了。

我看到这条头一个反应不是”哦又一个功能”，是”这下动到我那条抓网页的老链路了”。

浏览器住进了 Claude Code

先说这功能是啥。

Claude Code 桌面版新增了一个应用内浏览器。说白了，就是 agent 干活的时候，能直接在这个应用里开个网页看内容，不用你再单独给它接一套抓网页的工具。

以前想让 agent 读网页，你得自己搭。要么给它配 Playwright 这种浏览器自动化，要么挂个 MCP server 当中间层。这层东西不算难，但要装、要调、要维护，是笔碎账。现在这层被官方收进桌面版里了。

我一看就懂这是冲着什么来的。让 agent 能自己看网页，是它从”只会读你喂的文字”往”能自己去现场核对”迈的一大步。以前它写东西全靠你把料塞到嘴边，现在它能自己推门进去看一眼。对我这种拿 agent 天天产内容的人，这一步不小。

我一直在给 agent 喂网页

看到这我心里咯噔一下，因为这活我天天在干。

我有个 content-researcher，专门给写稿的 agent 找料。它的核心动作之一就是抓网页：先拿 playwright-fetch 去取页面正文，取不到再退到 r.jina.ai 这个兜底通道。这里头的坑不少。有的站直接封 UA，有的挂了重人机验证，playwright-fetch 硬闯经常空手回来，我才加了 r.jina.ai 当另一条腿，让它把页面先渲染成干净文本再吐给我，闯不过去的站靠它兜住大半。就这么一来一回调，我才让这条抓取链在大部分站上不掉链子。

我优先给 Claude Code 装的三个 skill 里，也有一个 /browse，图的就是它能真去页面上看一眼，而不是凭想象。做 GEO 那阵子我盯爬虫周报，天天要 agent 去某个页面确认它到底被抓成什么样：哪篇文章被 AI 爬虫抓了多少次、抓走的是正文还是一堆 JS/CSS。有回我发现抓取榜前十里七条是静态资源、真文章只占一条，就是这么盯出来的。所以浏览器对 agent 有多要紧，我算是有体感。

所以”让 agent 能看网页”这件事，我早就在干了，它是我流水线里一根天天在用的管子。

省掉的是自搭那层苦

这么一对照，桌面版内置浏览器省的是什么就清楚了。

它省的是我自己搭、自己维护那层抓取工具的苦。以前 playwright-fetch 偶尔抽风、r.jina.ai 偶尔限流，我都得回头查。要是官方这个内置浏览器足够稳，这类碎活能少一截。

说少一截我是真盼着。我这条流水线无人值守跑的时候，最烦的就是这种中间件半夜抽风：抓取失败了不报错，稿子就少一段料，我隔天才发现。我另一条部署流水线就出过这事，令牌过期了构建全绿，只有最后一步推送红了，静默挂了一整天没人知道。抓网页这层要是官方兜住了、稳了，我能少配一个”它又坏了”的告警。

这跟之前 X 出官方 hosted MCP 是一个路子：把你本来要自己搭服务器那层，收成平台自带的能力。对重度用 agent 的人，比多一个功能更值钱的，是少维护一层容易坏的中间件。

但省事的另一头，我心里也悬着一件事。

我担心的是它乱点

我担心的是，浏览器一旦住进 agent 手里，它能点的就不只是”看”了。

看网页是只读的，翻页、读正文，点错也没什么后果。可浏览器里还有一堆按钮：提交表单、确认下单、点”删除”。agent 在页面上自己动起来之后，这些撤不回的动作离它就一步之遥。

我给 agent 放权，一直守着一条土规矩：能撤回的活放心交，撤不回的活自己按。改代码有 git diff 兜底，我敢让它跑；可发东西、付钱、删数据这种一锤子买卖，那最后一下必须我自己点。浏览器把 agent 的手伸到了真实网页上，这条线就得盯得更紧。今天还有条新闻是 GPT-5.6 的 agent 把人家整个硬盘删了，这种事看着离谱，根子都在同一处：不可逆的操作太早交给了还会犯浑的 agent。

我给 agent 划权，一直按”点错能不能收回来”分两块。读得到、查得到的活放权彻底：拉日志、算数、找异常、只读巡检，这些现在就敢全交。会改线上状态、碰凭证、会发出去的活，收紧缰绳，宁可它停下问我一句。我用 coding agent 改真实仓库敢放手，靠的是它每件事都落在 git diff 里，看得见、回得去；可”合进主分支”那一下，我必须自己过手才点。运维那边更薄，没有 git diff、没有一键 revert，域名解析切错站几分钟就打不开，我吃过这亏。浏览器把 agent 领进真实网页，等于又给它开了一扇能”动”的门，我自然更得看紧。

所以这个内置浏览器我会用，但我大概率只敢先让它干”看”的活，”动”的活还得攥在自己手里，等它稳几个月、我摸清它的脾气再说。

agent 直连网页要来了

往大了看，这一步其实是个信号。

以前网页是给人看的，agent 要用得靠模拟人去点。现在两头都在往一起凑：一边是 Claude Code 这种把浏览器收进来、让 agent 直接开网页；另一边是 WebMCP 这类新标准，让网站主动把自己的功能暴露成 agent 能调的工具。中间那层”模拟人点来点去”的笨办法，早晚会被磨掉。

到那天，”网站怎么让 agent 更好地用”会变成一门新的功课，有点像今天大家做 SEO、做 GEO。我做内容分发时已经在琢磨怎么让 AI 引擎更愿意引用我，往后可能还得琢磨怎么让 agent 更顺手地用我的页面。

这事对做内容的人不算小变化。我博客做 GEO 那套：放开 AI 爬虫准入、给站点加机器能读的说明、每篇配结构化数据、盯着周报看谁来抓，本质就是把网页收拾成机器友好的样子。往后网页的头一个读者，说不定就轮到替人跑腿的 agent 了。我给 agent 写了大半年文档才反应过来，读它的早不是人；网页这边，同样这天迟早要到。

这功能刚上，我还没正经拿它跑过一轮活，上面这些是我对着自己那条老链路的头一反应。我最想验的是两件事：它抓取稳不稳、成功率比我那套 playwright 加 r.jina.ai 的土办法高多少。要是真能少配一个告警、少半夜爬起来查一次，我立马把老链路换了，回头再来摆效果。

想问问你：你会放心让 agent 在浏览器里自己点吗，还是跟我一样，只敢让它”看”、不敢让它”动”？评论区聊聊。
