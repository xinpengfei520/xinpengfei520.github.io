三个模型接力跑 16 小时，我那条流水线最怕的是交接掉链子

发表于 2026-07-14 更新于 2026-07-18 分类于 观点 阅读次数： 本文字数： 2.5k 阅读时长 ≈ 2 分钟
卡兹克晒出每天 Vibe Coding 16 小时的流程：Fable 5 出初版、GPT-5.6 Sol 审查纠错、Codex 目标模式最长连跑 17 小时。我对照自己那条十几个 skill 的流水线，结论是多模型协作真正难的不在谁最强，在按错代价乘频次乘单价把活分对、在每个交接口堵住 agent 自作主张。

哈喽，我是飞飞。

先说清楚，这篇不聊哪个模型最强。

我前阵子写过一串单模型的账。Claude 那两种省钱模式、GLM-5.2 该迁哪类活、最强的 Fable 5 我盯着价格又缩了手。那些讲的都是一个模型该不该用。今天这篇不一样，讲的是好几个模型加好几个工具，怎么在一条能自己连轴转的流水线里分工、接力、别掉链子。

起因是卡兹克晒的一套流程。他说自己每天 Vibe Coding 差不多 16 小时，干活分三步：Fable 5 出方案初版，他夸这一步「当世独一档」；GPT-5.6 Sol 审查纠错、优化方案；最后丢进 Codex 开「目标模式」全自动执行，据他自述最长连续跑过 17 小时。

我盯着这套分工看了半天。说实话，越看越眼熟。

他这三步，跟我那条写作流水线踩过的坑，几乎是一条道上的。先把话撂这儿：一条能连轴转 16 小时的流水线，真正难的不在找到那个最强的模型，在把每类活交给对的那个，让它们接力别掉链子。下面这几件事，是我拿自己那套对照他这套抠出来的。

我那把过秤的尺子

先说我怎么给活分模型。

我一开始不是这样的。刚上手那阵，逮着最强的 Opus 就用，什么活都塞给它。月底账单一出来，肉疼。那次我才逼自己改，按活分档，手动切。

现在我派一个 agent 干正经活之前，都会在心里过一遍秤。这活错一点，代价大不大？一天要跑多少次？用这个档，单价多少？三样一乘，答案就出来了。

错代价大又不常跑的活，比如一次关键的大重构，我舍得上最贵那个。质量要求没那么顶、又天天跑的碎活，比如批量改文件名、跑个格式转换，手动切便宜档，省额度。

这把尺子不玄。你把手头每个活拿出来，按这三样过一遍，八成的分工就定了。

当然它也有软肋。这尺子长在我手感上，状态好时分得准，忙起来就成了筛子。赶稿那阵我干过蠢事：一堆便宜档就能收的杂活，随手丢给了最贵那个；又把一个有点难度的活压给便宜模型，结果它答岔了，还得回炉重跑。说白了，我把「这是什么活」和「该走哪个档」糊成了一次拍脑袋。卡兹克把这两步拆开、各自定死，恰恰是我人肉干得最不稳的地方。

强模型只在硬活上烧

卡兹克把出初版这一步交给 Fable 5，我很认。

Fable 5 是现在全球主流里最贵的那档，输入输出的价压得人心里打鼓。但它有它的道理。出一个大方案的初版，最考验的是从零到一那口气，判断力得顶。这种活错一步，后面全歪，属于错代价最大的那类。

放在我的尺子里，这正是该上最贵档的地方。

反过来，我绝不拿 Fable 5 去干杂活。把一批文件名统一改个格式还让它上，纯属烧钱。这种活对错一眼能看出来，便宜档一次就过。

一句话，最贵那个只在「错一点就要命」的活上烧。别的活，越便宜越好。这跟你月底账单直接挂钩。

审查这一环最不能省

卡兹克第二步用 Sol 审查纠错，这一步我盯得最紧。

因为我踩过它的坑。

我拿 agent 把一个 7000 多行的老 Java 项目迁成 Kotlin，一天迁完上了线。听着挺爽。可过程里它干了几件让我后背发凉的事：把一处空指针判断顺手「优化」掉了，把一段分支逻辑悄悄改岔了，还有个字段名凭空消失。编译过，跑也过，满屏 diff 我一眼看不出来。

最后能捞回来，靠的是流水线里专门设的一道对抗校验。我在执行和交付之间，插了一组专门唱反调的 agent，默认就是来找茬的。它一报错，我才发现那几处被改坏的地方。

所以卡兹克让 Sol 专门审 Fable 5 的产出，绝不是多此一举。长任务最怕的是「假装跑完」，前面 190 步都对，第 191 步偷偷写坏一个字段，面板照样全绿给你看。审查这一环，就是专门堵这个的。谁出的活谁自己审，等于没审。

接力处最容易掉链子

三步里我最担心的，是交接那一下。

模型再强，接力棒没递明白，照样翻车。我这条教训是拿配图 skill 换来的。

我那个配图 skill，早期没把话说死。它该干的活是给正文写个「截图 N」的占位，等后面另一个环节来替换成真图。可我没写死「只许写占位、绝不许自己上传或塞链接」这句。结果它自作主张，把图床链接直接塞进正文，还漏了图。

后来我把这条抠成硬规则，一个字都不留活口，它才老实。

交接口就是流水线最脆的地方。我现在的做法，是给每个环节配一张「岗位接口卡」：进来什么、出去什么、边界在哪，全写死。新环节上工先对着这张卡认输入输出，别让它自由发挥。

还有更阴的。全自动流水线最贵的坑，是没人盯着的时候，失败会悄无声息地堆。我那个博客镜像站部署过一次，构建全绿，只有最后一步 git push 因为 token 过期认证失败，静默挂了一整天没人发现。从那以后，我给「失败」本身单配一个告警，直接推到手机。链路可以无人值守，但它出事得有人立刻知道。

目标模式我还没敢放

聊到这，得说句老实话。

卡兹克那套 16 小时连轴转、Codex 目标模式最长跑 17 小时，我没照搬过。我甚至没正经跑过目标模式连跑这么久。

我十几个 skill，九成还是跑「回合循环」，干一轮就停下等我验一眼。官方把 agent 循环分四档，回合、目标、时间、主动，越往后越放手。我目标循环一直没敢正经上，原因很简单：我写文章这活，「算干完了」的标准是软的，没有一条机器能替我判死。测试跑绿是硬标准，一篇文章好不好没这种东西。

标准是软的，就没法真放手让它自己跑到「完成」。这跟 Codex 行不行没关系，是我这类活还没到能全托的时候。

有意思的是，OpenAI 把我在终端手动干的这些事，切模型、调 skill、起后台 agent，包装成了界面上人人能点的模式切换。方便是真方便。可正因为我手动趟过这些坑，让它无人盯着跑几个小时，我心里那道闸反而绷得更紧。

这套分工也不是我一开始就想周全的，是一个个坑踩出来的。我专门做了个 skill，每写完一篇就把新踩的坑自动提炼成规则，回写进那份规则手册，攒到现在一百多次迭代。skill 每被用一次，就收紧一点。这才是我敢让流水线多放一点手的底气。

给你一个今天就能改的最小动作：把你手头那条流程，拆成出初版、审查纠错、自动执行三段，每一段定死交给谁，别让一个模型从头干到尾。再在执行和交付之间，插一道哪怕很糙的检查。

你那条流水线，最容易掉链子的是哪一环？哪个模型你到现在只敢让它干某一类活、别的碰都不敢碰？评论区聊聊。
