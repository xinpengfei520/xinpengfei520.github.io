Claude 住进了你的 Slack：@一下就接活的 AI 同事，你敢交多少活给它

发表于 2026-06-29 更新于 2026-07-01 分类于 观点 阅读次数： 本文字数： 3k 阅读时长 ≈ 3 分钟
Anthropic 在 2026-06-23 发了 Claude Tag，把 Claude 做成住在 Slack 频道里的共享 AI 同事，@一下就接活，跑在 Opus 4.8，对 Enterprise 和 Team 开放 beta。真正变的地方是 AI 从单聊工具挪进了团队群，成了能接力干活的同事；但它替你跑完的是产出，合不合、发不发的判断还落在人头上。

哈喽，我是飞飞。

这两天我刷推，看到 Anthropic 发了个叫 Claude Tag 的东西。第一眼我以为又是个能在 Slack 里 @ 的机器人，没当回事。点进官方博客读完才反应过来，这事比我想的大。它要干的，是把 Claude 直接拉进你团队的群里，当一个能接活、能记事、还会自己冒头说话的同事，而不只是多给你一个问问题的窗口。AI 从「另开标签页单聊」，挪进了你工作真正发生的那个群。

发布时间是 6 月 23 号，今天起对 Claude Enterprise 和 Team 用户开 beta，跑在 Opus 4.8 上。旧的那个 Claude in Slack 应用 8 月 3 号下线，管理员 30 天内可以迁过去。

Claude 住进了 Slack

用法其实你很熟。在一个 Slack 频道里 @Claude，跟它说句人话提个需求，它会把任务拆成几个阶段，挨个用手里的工具去做，干完在 thread 里回你产出。这套路子，用过 Claude Code 的人闭着眼都会。

官方给了个数字我盯着看了半天：他们自己产品团队 65% 的代码，是内部版的 Claude Tag 写的。原话是「65% of our product team’s code is created by our internal version of Claude Tag」。这话的分量不在「会写代码」，在「已经长进日常」。Anthropic 说现在 @Claude 是他们干活的主要方式之一，连追产品数据、处理工单、查 bug 根因都在 @ 它。

它跟单聊差在哪

真正新的东西不是「能聊」，聊天框早就有了。说白了，差别在三处。

头一个是多人共享。一个频道里只有一个 Claude，所有人对着同一个它说话。你同事昨天 @ 它干了一半的活，你今天接着 @ 一句就能往下推，不用从头跟它解释。这跟你一个人开个对话框单干，体感完全是两码事，更像旁边真坐了个同事。

它还会攒上下文。跟着频道待久了，它对这摊活越来越熟，你不用一遍遍重复背景。授权了它还能从别的频道和数据源学。这里官方划了条线我觉得值得记一下：「It doesn’t report from private channels.」私有频道它不读、也不往外抖。

最有体感的是它会主动。开了 ambient 模式，它会自己冒头，flag 它觉得你该知道的事，追那些没人收尾、晾在那儿的 thread。再加上异步：派完活你忙别的，它能给自己排期，一个项目连着跑几个钟头甚至几天。Anthropic 自己说，他们现在大把时间花在「同时 delegate 给一群 Claude」。

工作方式从「我用一个工具」变成了「我管一队 Claude」。

这步棋冲的是上下文

把这事往大了看，它不只是个提效功能。TechCrunch 的判断挺直接，说这是一步「strategic play to capture organizational context」，冲着抢企业的组织上下文去的。

这个我认。一家公司最值钱的东西，很少是某段具体代码。更多是那些散在聊天记录、文档、工单里、从来没人专门写下来的隐性知识：我们这儿到底是怎么干活的。谁先把这套东西喂进自己的模型，谁就拿到最深的锁。配套有个数据能佐证势头：Ramp 的 2026 年 5 月 AI Index，覆盖五万多家美国公司，Anthropic 的企业采用率到了 34.4%，第一次反超 OpenAI 的 32.3%，主要推手就是 Claude Code。Claude Tag 是顺着这股劲往里再扎一层。

有意思的是 Slack 是 Salesforce 家的。The Information 报道说，Anthropic 这个产品一发，Salesforce 内部有些员工有点懵，觉得像被「入侵」了自己的地盘，可 Salesforce 官方又在社媒帮着推。这种别扭，恰好说明这块阵地有多关键。

我最担心静默失败

泼点冷水。我不是 Enterprise 用户，没在公司频道里真跑过 Claude Tag，下面这些是我拿自己天天用 Claude Code 的体感往这上面套，你听个参考。

我最先担心的，不是它会不会写错代码。代码写错你还能 review 出来。真正让我犯怵的是另一头：它在后台默默干活、没人盯着的时候出了岔子，谁知道。

这事我真栽过。我博客 vance.xin 整条发布是 GitHub Actions 自动化的，push 上去自动构建部署，workflow 里还自挂了两段，每次部署把最新链接推给百度收录和 IndexNow。前阵子有个部署用的 token 过期了，构建一直显示成功，只有最后一步推送悄悄红了。没人盯 workflow 状态，镜像站就这么挂了一整天，我是隔天主动去翻才发现。主站靠另一条独立的 Docker 流水线还活着，镜像站却空了一天没人知道。自动化最贵的那笔成本，藏在没人盯的时候：失败不会喊，它只会悄没声地堆。

Claude Tag 的异步加 ambient，刚好把这个风险放大了。它能自己连干几天，能主动往群里发东西。爽是真爽，可它替你跑完的是产出，不是判断。那个 PR 该不该合、那条结论敢不敢往外发，没人能替你点头。越敢把活交出去，越得在背后留一道「人能看见」的闸。

后台烧钱要先过秤

还有一笔账是钱。Claude Tag 跑在 Opus 4.8 上，token 花费记在公司头上，官方允许按组织和频道设上限。

这条我体感很深。我每个月给 Anthropic 的钱不算少，让 agent 干正经活之前，我都会下意识在心里掂一下值不值得动 Opus，杂活就手动切便宜模型省额度。这种「用之前先过个秤」的绷着感，就是「贵」最真实的样子。换到一个能自己排期、连轴转几天的 Claude 身上，这根弦只会更紧。那个按频道、按组织设的 token 上限别嫌烦，它是帮你在睡觉的时候也替你守着钱包的。

好在 Anthropic 把治理这块做得算克制：memory 按频道隔离，销售用的 Claude 不会把记忆漏给工程那边；私有频道不读；花费有两级上限；还留了一份完整日志，谁派了什么活都查得到。这些兜底是必要的。但说句实话，把全公司的上下文交给一家供应商，供应商锁定、数据隐私、单点依赖这些事，技术兜得住一部分，剩下的得靠你自己判断。

锁定这件事我也有体感。我光写文章这一条流水线，就挂着十几个 skill：拉资讯的、查重的、配图的、发公众号的，每一个都是照着 Claude Code 读 skill、认 frontmatter、调 hook 的脾气一点点磨出来的。真要把模型底座换成别家，这些 skill 调工具吐出来的格式十有八九会飘，得挨个重新校。光是把那套专门防 AI 味的 grep 规则重新调顺，就够搭进去好几个晚上。这部分时间成本，省模型钱那本账上从来不记。

我这才十几个 skill，就已经被黏得挺死。一个把组织上下文喂了好几个月的公司频道，想搬家是什么概念，你自己掂量。

哪些活敢交哪些不敢

绕一圈回到开头。Claude Tag 现在还是 Enterprise 和 Team 才有，但这个形态会往下走，迟早轮到个人和小团队。

它确实把一件事往前推了一大步：AI 已经从你单独打开的那个对话框，挪进了你团队干活的那个群。这一步我乐见。我自己十几个 skill 全长在 Claude Code 上，一天到晚在跟它配合，太知道「它在你身边接活」和「你专门去找它问」差在哪。

可越是这样，那道题越躲不开：哪些活你敢直接交给群里的 AI，哪些你非得自己捋一遍才敢点确认。你那道兜底的闸，到底想划在哪一步。这个问题我自己也还没想全。你呢，要是你们群里明天进来一个 @Claude，你第一个敢丢给它的活是什么，又有哪件你打死也想自己盯着？评论区聊聊。
