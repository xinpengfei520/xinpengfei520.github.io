AI 自己提工单、揪出工程师漏答：我手动盯的那套运维，它一晚跑完了

发表于 2026-07-03 更新于 2026-07-14 分类于 观点 阅读次数： 本文字数： 2.7k 阅读时长 ≈ 2 分钟
卡兹克让 Claude Fable 5 自主优化 AIHOT 网站 GEO：模型自己起 22 个 agent 调研、提工单、揪出工程师漏答、发现漏洞自加暗号，一整条运维链自主跑完。对照我手动盯博客 GEO 一个多月、每周手算爬虫抓取 5942 次再手改 robots，真正的变化不在 AI 会写代码，在运维正从人盯挪向人审：只读巡检可现在就交，碰线上状态和凭证的活还得自己按确定键。

哈喽，我是飞飞。

昨天刷到卡兹克一篇全记录，我盯着看了好一会儿。他让 Claude Fable 5 去优化 AIHOT 那个网站的 SEO 和 GEO，就是我天天拉 AI 资讯的那个站。Fable 5 这回干的活，比写一段代码大多了。它把一整条运维链，自己从头到尾跑完了。

我最近连写了几篇模型成本、选型的账，还有一串 Claude Code 的功能。这篇跟那些都不搭界。它戳中的是另一条线，一条我一直在手动盯、却一篇没写过的线：给自己博客做 GEO 的日常运维。看完我没顾上夸它强。反倒有点发怵，因为它干的这套活，我正一样样手动在做。

先看它一晚干了啥

先把卡兹克记的这条链摊开。这些数字和情节都是他记录的，我没亲手跑过，先说清楚。

Fable 5 一上来自己起了 22 个 agent，花约 40 分钟调研，揪出「豆包 App 每天六千多次访问没被统计」这种藏得挺深的异常。

规划境外加速的时候，它把 Opus 4.8 先提的 Cloudflare 方案给否了。理由很具体：这方案没法做到国内直连、国外分流，而且 Cloudflare 从 2025 年 7 月起就默认拦 AI 爬虫了。它转头改用火山引擎的 CDN。

接下来更让我意外。开火山引擎的 CDN 需要走白名单，它自己找到工单入口，提了一个写得挺专业的工单，22 分钟就开通了。开通过程里，它发现对接的工程师漏答了回源 IP 网段的问题，礼貌地追了一句，还顺手补了个备选方案。它甚至看出官方给的方案有个安全口子，自己加了道暗号验证补上。

当晚 23:30 它切了域名解析，10 分钟后，616 个海外请求走上了新线路。收尾时它生成了一份运维文档，还特意提醒边缘证书 10 月 2 号到期、附上了续期步骤。

一条从调研到收尾的完整运维链，它一个人扛完了。

它干的是我手动的活

我坐不住，跟它会不会写代码没什么关系。代码会写这件事，早不新鲜了。

真正让我坐不住的，不在它写得了代码，在它把「起 agent 调研、定位异常、比选方案、提工单、追着人要答案、切换上线、留下文档」这一整条链，自己一环扣一环走完了。这几件事拆开看，恰好是我这一个多月一直在手动做的 GEO 运维。

一个能自己判断该找谁、该问什么、发现漏洞还自己补的东西，跟一个等你派活的工具，是两种感觉。

我盯 GEO 那套笨功夫

说回我自己这本账。我给博客做 GEO 大概一个多月了，一直是手动加半自动，土得很。

我确实写了个 GitHub Action，每周一早上 9 点自动生成一份「AI 爬虫周报」的 issue。但它只管生成，不管分析。真正费神的活全在我手上。

每周一我得自己动手：把最新那份周报拉下来，把十六种 AI 爬虫的访问数一项项相加，得出抓取合计；再把 chatgpt、perplexity、豆包、Kimi 这些来路的回流数相加；回流除以抓取算出回流率；最后在一个 markdown 表格里补上一行。

上周那行是这么写的：抓取 5942 次，比基线涨了 26%，可这增长几乎全靠 Meta 一家撑着，它的爬虫占了 69%；同期 GPTBot 从 1869 掉到 504，ClaudeBot 直接挂零；回流只有 3 次，还是个位数的噪声，国内引擎全是零。

我盯着这行数看出个毛病。抓取排前十的里头，有 7 条是 JS、CSS 这种静态资源，真正的文章就 1 条。爬虫把宝贵的抓取额度，全烧在没有引用价值的静态文件上了。所以 7 月 2 号，我手动去 robots.txt 里给 AI 爬虫那一组加了两行，把 /js/ 和 /css/ 拦掉，把抓取导回正文。

手动的活还不止这些。我每篇文章发到知乎，都会在文末手动贴一张带原文回链的作者卡，专门往知乎这种国内 AI 语料的富矿里，塞进带链接的内容。这条回链我上个月还栽过跟头。知乎版一直用英文 slug 拼链接，可我这站是老版 Hexo，permalink 只认数字文件名，那批回链就这么 404 了大半个月，我一直没发觉，直到 7 月 2 号改成数字 permalink 才修好。那阵子台账上国内引擎的回流一直挂零，多少跟这条断掉的链有关系。你看，连一条链接对不对，都得靠我自己隔段时间去点一遍。

这套活不难，就是琐碎，得有个人一直盯着。而 Fable 5 一晚上自己跑完的那条链，拆开看，跟我每周一手动补台账、手动改 robots、手动补回链的动作，是同一类活。

从人盯变成人审

把这两件事摆到一块儿，我咂摸出的其实是运维这活正在悄悄换一种做法。

过去我做 GEO，核心动作就一个字，盯。盯周报、盯哪家爬虫在涨、盯哪条回流是噪声、盯前十里混进来的静态资源。我是那个从头盯到尾、还得自己动手的人。

说白了，我这套台账的本质，就是拿人肉在做一次轮询。每周一被日程提醒叫起来，手动去查一遍站的状态，再手动决定这周该改哪儿。这个轮询的频率是我定的，中间但凡哪周我偷了懒没补，这条链就断在那儿，没人接。

Fable 5 那套演示里，人往后退了一步。它自己盯、自己动手、自己写文档，卡兹克要做的，从「自己盯着改」变成了「审它交上来的结论和那份运维文档」。

它省下的，不光是手上那点操作，是把「一直得有个人盯着」这件事本身给免了。运维正从人盯，挪向人审。这才是让我真正坐不住的地方。

哪步我还不敢撒手

但我没打算明天就把 robots.txt 交给一个 agent 去自主改。

说实话，它能自己提工单、自己切域名解析，这几样都是会改状态、会带来不可逆后果的操作。域名解析切错一次，站可能几分钟打不开；robots 写错一行，可能把本该放进来的检索爬虫也一起拦在门外，而这种错，我大概率要等一周后看周报才发现。

代码域里我还有 git diff、能一键 revert；运维域这层安全网，薄得多。

所以我给自己划了条线。只读的、查得到出处的活，比如拉周报、算数、找异常、给结论，我乐意现在就交出去。会改线上状态、碰凭证、碰解析的活，我还想自己按那个确定键。我认它能跑完整条链，但那道「要不要真按下去」的闸，眼下还得攥在我手里。

今天先交一段只读的活

这事对同样天天盯着自己站的你，其实有个特别小、今天就能上手的动作。

别一上来就让 agent 自主去改配置。先把一段只读的运维巡检交给它：让它替你拉一遍日志、算一算这周哪个爬虫在涨、哪些请求是白烧的静态资源，然后只给你一份结论。你要做的，从「自己趴日志里扒」变成「审它这份结论准不准」。

就这一步，你先把「人盯」里最枯燥的那一截交出去，又没把「人审」那道闸松掉。等你审了几回、它的结论回回都对得上，再考虑要不要往前放一格。

说回来问你一句：你手上那些运维和账号操作里，哪一段你乐意现在就让 agent 自主跑？又是哪一段，你打死都要自己按确定键？评论区聊聊，我挺想知道大家那道闸，都划在了哪一步。
