X 官方 hosted MCP 上线：agent 直连 X，省掉你自己搭服务器那层苦

发表于 2026-07-01 更新于 2026-07-11 分类于 观点 阅读次数： 本文字数： 2.6k 阅读时长 ≈ 2 分钟
2026-06-30 X 官宣 hosted MCP server，Claude Code、Cursor、Grok 等任何 MCP 工具零配置直连 X API。它不加新能力，只是把你自己搭 MCP、host、连 API、处理 OAuth 那整层脏活替你包圆了。定价按次无月费，读一条帖约半分钱、发帖查用户约一分钱。信号是 MCP 正在成为 agent 直连各家 API 的通用插头，平台不进 MCP 就等于对 AI 助手隐形。

哈喽，我是飞飞。

6 月 30 号，X 官方开发者账号发了条不长的推：hosted X MCP 上线了，agent 现在能直连全世界最好的实时信息源，Grok、Cursor 或者任何 MCP 工具零配置就能接。马斯克转手转发，配了三个字：试试看。

这条推底下不算热闹，但我看到的时候坐直了一下。因为这事戳的，正好是我天天在干、也天天在被烦的一件事：怎么把外部工具接进我的 agent。

最近我写的是怎么写 skill、怎么挑模型，聊的都是 agent 内部怎么搭。这篇换个层面，聊 agent 怎么伸出手去够外面的世界。

X 给 agent 开了直连口

先说这事本身是什么。

以前你想让 Claude、Cursor 这类 AI 助手读 X 上的实时数据，得自己搭一整套东西：写一个 MCP server、找地方把它 host 起来、对接 X 的 API、还得处理 OAuth 认证和限流。一层一层，全是脏活累活。

现在 X 官方把这套自己托管了。你不用搭服务器，地址就是 api.x.com/mcp，走 xurl 授权，200 多个接口摆在那儿，搜帖、看趋势、翻书签、查用户、分析对话，接上就能用。

TechCrunch 说得很实在：它没给你任何新能力。X 的 API 早就能干这些，hosted MCP 只是把这些能力更省事地端到 AI 面前。

它省掉的到底是什么

那它到底省了啥？我得先讲清楚，这才是重点。

它真正的价值，不在给了 agent 什么新本事，在它把「你自己搭 MCP server、自己 host、自己连 API、自己处理认证」那整层苦活替你包圆了。

打个比方。以前是你想喝水，得自己打井、铺管、装水表；现在人家把水龙头装到你墙上了，一拧就出。水还是那口井的水，但中间那段脏累活没了。

对偶尔用一次的人，这点省事无所谓。可对天天要把七八个工具接进 agent 的人，省掉的这层，就是每天最磨人的那部分。

我自己搭 MCP 的苦

这层苦，说真的，我是真尝过的。

我这个号背后挂着十几个 skill，全靠 Claude Code 串起来跑，里头接了一堆 MCP：配图用 banana、搜资料用 brave-search、一文多发用另一个。这些 MCP，就是我 agent 伸出去够外部世界的那些手。说个近的，你现在看到的这篇文章，封面是我让 banana 那个 MCP 出的，写之前的资料也是 brave-search 那个 MCP 一条条搜回来的。它们顺的时候，我几乎感觉不到它们存在，一句话的事。

可它们不顺的时候，那才是真磨人。最折腾的一次，是我自己 build 了一个同步发文的 MCP 接进 Claude Code。官方那个版本不稳，我干脆自己打了个保活补丁塞进去。最要命的是它底层那个 Service Worker，跑着跑着就自己断联了，agent 这头还在吭哧吭哧派活，那头其实早没人接了，任务就那么静静地悬在半空，也不报错。

我一开始压根没往这上面想，查了半天日志才反应过来是它睡过去了。那阵子我就反复地调、反复地试，塞各种办法把它踢醒、让它别掉线，前后耗了好几个晚上，最后才勉强让它稳住跑通。

那种苦你搭过就懂：模型多聪明都没用，真正卡你的永远是接入这一层。认证、保活、断线重连，全是不出彩、但一步都少不了的活。X 这个 hosted MCP，等于把这类脏活官方接管了，我看着是真有点羡慕。

这笔账怎么按次算

省事归省事，钱这块得单独算笔账。

它是纯按次收费，没有月费。X 的 API 从今年 1 月就转成了这种计价：读一条帖大概半分钱，发帖或者查一个用户大概一分钱。有人实测拉了近三天的书签，一共花了一毛钱左右。

单看很便宜，几乎等于白送。但你要真把它挂上一条自动化流水线，让 agent 没日没夜地搜、翻、查，这账就得自己盯着了。按次计费最怕的就是 agent 空转，一个没写好的循环，可能半夜替你烧掉一把零钱还没人发现。

这方面我是有点应激的。我天天用 Claude Code，每个月给模型交的钱不算少，所以现在动那些贵档模型之前，我都会先在心里过一遍秤：这一趟值不值得让它跑。按次收费的 API 也一样，好用是好用，但每一次调用背后都是真金白银在走。

所以接是可以接，我的习惯是先给它设个用量的心理红线，别让它撒开了跑。真上流水线之前，我一定会先小规模试几天，把大概每天要花多少心里有个数，再决定放不放开。

只读还是给写权限

还有件事我得提醒你，别被「零配置」冲昏头。

它是用你已有的账号权限连的。也就是说，你授权给 agent 的那一刻，等于把「能替你读什么、甚至替你干什么」这件事，交到了一句配置里。

这里我特意查了一下，各家说法还不太一致。有的媒体明确说它目前只读、不能自主发帖；也有人演示从编辑器里直接发文章。到底放开到哪一步，我暂时按保守的理解：以读取实时数据为主。

我的态度很清楚：读没问题，让 agent 替我搜帖、看趋势，风险可控。但说白了，「替我发帖」是完全另一个风险等级的事。接入越简单，越要在放权限这一步上多想两秒。方便和风险，从来是同一枚硬币的两面。

MCP 成了通用插头

把镜头拉远一点，X 这步棋就更有意思了。

它不是一家在动。GitHub、Slack、Notion、Stripe、Salesforce 早就上了官方 MCP，现在 X 也进来了。MCP 这个协议，正在变成 agent 连接各家服务的那个通用插头，有人管它叫 agent 世界的 USB-C。

去年年底，Anthropic 干脆把 MCP 捐给了 Linux 基金会，让它不再属于某一家、变成中立的公共标准。现在五大 AI 厂商全在支持它。

这背后是个挺硬的信号：平台们开始为 agent 而不只是为人建东西了。往后哪个平台不接进 MCP，就等于在 AI 助手眼里隐了形，agent 想用你、却够不着你。X 这步与其说是给开发者行方便，不如说是抢在下一轮之前，先把自己插进每一条 agent 工作流里。

我准备怎么接一个

说回我自己。

老实交代，X 这个 hosted MCP 我还没真接过，上面聊的是它的机制加我自己搭 MCP 的体感对照。但我已经动心了。

我准备找个空档，先只读地接一个进 Claude Code，让它帮我搜搜某个话题下的实时讨论、看看趋势，写权限一律先不给。等真跑过一阵，攒下第一手体感，再回来跟你聊它到底顺不顺、那笔按次的账实际怎么样。

接入这层越来越不设防，是好事，也逼着我们每个人重新想一个问题：你到底愿意让 agent 替你够到哪些东西。

所以想听听你的：如果让你给自己的 agent 接一个直连外部的口子，你会先接谁的 API？只给读，还是也敢给写？有没有被自己搭 MCP、配认证坑到过的经历？评论区聊聊，让我看看不是一个人在折腾。
