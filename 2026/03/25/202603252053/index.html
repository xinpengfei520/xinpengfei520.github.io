Claude Code 最能打的 10 个 Skill：装上这些，效率直接上一个台阶

发表于 2026-03-25 更新于 2026-04-21 分类于 工具 阅读次数： 本文字数： 3.1k 阅读时长 ≈ 3 分钟

Claude Code 用了一段时间之后，你会发现一件事：原生的 Claude Code 只是个底座。真正让它变强的，是 Skill。

Skill 就是给 Claude Code 装插件。每个 Skill 是一段预设的工作流程，用 /skill名称 呼出，Claude 立刻知道该怎么干。不用每次写长 prompt，不用反复解释你的需求，一个命令，一套完整的专业流程就跑起来了。

我用下来觉得最值钱的有 10 个，按实用程度排下来，跟你分享一下。

1. /browse — 让 Claude 真的会「看」网页

这是我用得最高频的一个 Skill。

/browse 给 Claude Code 接上了一个无头浏览器。你说”打开这个页面截图”，它真的去打开，截图发给你；你说”测试一下这个按钮点了有没有反应”，它真的去点。不是模拟，是真实的浏览器操作。

场景很多：验证部署是否成功、测试一个表单有没有 bug、对比改版前后的页面长什么样、拿竞品截图做参考。以前这些事要自己手动做，现在丢给 Claude，它跑完把结果发给你。

2. /find-skills — 让 Claude 帮你找 Skill

有点套娃，但很实用。

当你知道自己想做某件事，但不确定有没有现成的 Skill 时，用 /find-skills。你描述需求，Claude 去搜索已安装的 Skill 列表，给你推荐最合适的那个，还会告诉你怎么触发它。

我刚开始用 Skill 的时候，光是记住哪个 Skill 叫什么名字就够呛，后来发现有需求直接问 /find-skills 就行，它比我自己翻文档快多了。

3. /skill-creator — 自己造 Skill

用一段时间之后，你会发现有些工作流是你自己独有的，市面上找不到对应的 Skill。这时候 /skill-creator 就派上用场了。

你描述你想要的工作流，/skill-creator 帮你生成一个新的 Skill 文件，保存到你的 .claude/skills/ 目录里。之后这个 Skill 就归你了，随时用。我自己用它创建了一个适配这个博客的 writing-agent，现在写文章就是一句话的事。

它还支持给 Skill 跑评测——对比两版 Skill 哪个效果更好，有点像 A/B Test，但是针对 AI 工作流的。

4. /investigate — 系统化的 Debug 流程

遇到 bug 最怕的不是 bug 本身，是绕了半天、最后发现治标不治本。/investigate 的设计逻辑很简单粗暴：不找到根因，不动代码。

它分四个阶段：调查现象 → 分析日志和代码 → 建立假设 → 实施修复。每个阶段都有结构化输出，Claude 不会跳步骤，不会因为看到一个可疑的地方就直接改。这对那种”改了 A 又冒出 B”的连锁 bug，特别有效。

我现在遇到说不清楚的 bug，第一反应就是 /investigate，而不是直接跟 Claude 说”帮我修这个错误”。

5. /qa — 测试 + 修 bug，一条龙

/qa 是我最喜欢的一个 Skill，因为它省了一大堆来回的对话。

你跑 /qa，Claude 会系统地测试你的 Web 应用：找 bug、截图记录、写报告，然后直接去修，修完再验证，验证通过才提交。不用你一个个告诉它该改哪里，它自己走完整个测试-修复-验证的循环。

有三个档位：Quick（只查高优先级问题）、Standard（加上中等问题）、Exhaustive（连视觉细节都查）。上周我上了一个新的评论组件，跑 Standard 直接抓出来一个 XSS 漏洞——用户名里的尖括号没转义，差点就带着这个上线了。

如果你只想要报告不想让它动代码，用 /qa-only，纯报告模式。

6. /review — 合代码前的最后一关

代码改完准备合并，跑一下 /review。

它会分析你的 diff，检查这几件事：SQL 有没有安全漏洞、有没有把不可信的输入直接传给 LLM、有没有条件竞争导致的副作用、还有其他常见的结构性问题。

不是风格检查，不是”这里可以写得更优雅”，是真正影响安全和正确性的问题。上次它抓出来一个我写的 SQL 拼接，我自己看了三遍都没注意到——就这种肉眼容易滑过去的问题，机器反而比人敏感。

7. /ship — 从 commit 到 PR，一步到位

/ship 把 git 工作流的几个步骤打包成一个命令：检测并合并 base 分支 → 跑测试 → review diff → 更新 CHANGELOG → 提交 → push → 创建 PR。

之前有一次发版忘了合 base 分支，PR 里多了一堆不相关的 diff，review 的人直接懵了。/ship 之后这种低级错误就没再出过，想发版了直接跑这一个命令，剩下的事 Claude 搞定。如果你用 /land-and-deploy，还能继续接上自动合 PR、等 CI、验证生产环境的流程。

8. /autoplan — 让 AI 帮你把方案想透再动手

这个 Skill 解决一个很常见的痛点：你把需求扔给 Claude，它马上开始写代码，但写到一半你发现方向就不对。

/autoplan 会先让 Claude 做完整的方案设计：从产品视角（这个功能值不值得做）、工程视角（架构怎么设计）、设计视角（用户体验有没有问题）三个角度全部过一遍，形成一份完整的执行方案，然后再问你确认。

不过说实话，小改动别用它——有一次我就改个按钮颜色，它也跑了完整的三轮评审，属于杀鸡用牛刀了。但凡涉及架构调整或者新功能，先跑 /autoplan 确实能少走弯路。

9. /git-commit — 永远不再为 commit message 发愁

技术上这个是最简单的 Skill，但我用得最高频之一。

/git-commit 读取你的 diff，分析改动内容，自动生成一条符合仓库风格的 commit message，然后精确暂存相关文件、提交。

最重要的一点：它只暂存和这次任务相关的文件，不会把其他改动一起带进去。这对同时在改多个东西的场景，特别关键。commit message 写得也很准——不是”update code”，是”fix(auth): prevent session token from persisting after logout”这种有信息量的描述。

10. /mcp-builder — 给 Claude Code 接入任何外部服务

这是一个偏进阶但天花板很高的 Skill。

MCP（Model Context Protocol）是 Claude 跟外部工具通信的标准协议。/mcp-builder 帮你从零开始构建一个 MCP Server——你想让 Claude Code 调用你们公司内部的 API？接入某个第三方服务？有了 MCP Server，Claude 就能直接用这些工具，就像它原生支持一样。

支持 Python（FastMCP）和 Node/TypeScript 两种写法，还有评估模块帮你测试 Server 的质量。我用它给博客接了个图片上传的 MCP Server，现在写文章时 Claude 可以直接帮我传图到 CDN，不用再手动操作七牛云后台了。如果你有类似的定制需求，这个 Skill 是最底层的入口。

这 10 个 Skill 里，如果你只装三个，我的建议是：/browse、/git-commit、/investigate。

/browse 让 Claude 能真正帮你测试页面，不只是聊代码；/git-commit 解决每天最高频的小摩擦；/investigate 在你最头疼的时候，让 debug 变得有章可循。

装 Skill 的方式很简单，在 Claude Code 里输入 /find-skills 搜索你想要的，或者去 GitHub 找社区分享的 Skill 直接复制到 .claude/skills/ 目录下就行。

用 Claude Code 用到现在，我越来越觉得——原生能力决定下限，Skill 决定上限。你有没有特别好用的 Skill，评论区说说？
