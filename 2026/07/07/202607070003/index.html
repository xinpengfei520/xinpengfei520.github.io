Claude 的设计大脑被逆向成 20 章开源，我对着自己那本规则书读完了

发表于 2026-07-07 更新于 2026-07-13 分类于 观点 阅读次数： 本文字数： 2.4k 阅读时长 ≈ 2 分钟
有人把 Anthropic 官方设计能力 Claude Design 的系统提示词逆向重建、以 MIT 开源，GitHub 上是 20 章提示词加 14 个技能。它真正的分量不在规则多，在把无障碍、AI 垃圾感、自主决策这些软要求全抠成了可执行的硬动作。它是照着输出反推的高置信度推断，离官方源码还差一层。

「哈喽，我是飞飞。」

我这两天在 GitHub 上翻到一个仓库，有人把 Claude 出设计稿背后那套系统提示词，一章一章逆向出来，做成了一个能直接拿去用的提示词加技能库，MIT 开源，一千两百多个 star。我盯着它的目录看了挺久。

先把丑话说前头。这是社区逆向重建出来的东西，跟 Anthropic 官方放出来的源提示词是两码事。仓库开头就写着 reverse-engineered，作者是照着 Claude 的输出反推的，顶多算高置信度的推断，离官方源码还差着一层。这个界我特别在意。前几天我刚逆向过 Claude Code 两回，一回是它偷偷把你输入的撇号换掉，一回是它二进制里暗藏一百多个域名。那两篇我抠的是它藏了什么、有没有碰你的隐私。这篇不一样，我想看的是一套顶级系统提示词，到底怎么把规则组织起来的。

我也维护一本规则书

我为什么对一份提示词这么上心？因为我自己天天在写这种东西。

我做了一整套写作 skill 串起来跑，其中管文风的那本规则书，从我头一回记下「别用中文破折号」到现在，已经被我喂到一百一十多次了。每写一篇踩个新坑，就回去加一条。旁边还另挂一本专管「怎么写好」的招式手册。主流程文件我尽量写薄，细节全塞进这两本外挂，用到哪本翻哪本。

所以我看这个逆向出来的仓库，一眼就认出门道了。二十章系统提示词加十四个技能，主提示词管大方向，技能按需调用，跟我这套分层是一个路子。说白了，一份系统提示词写到一定长度，你不可能把细节全堆在一页里，逼着模型每次都背一整本书，只能拆。它拆成二十章，我拆成两本外挂，动作一样。看到别人把同一个工程直觉做到这个规模，我有点亲切。

最狠的一章是无障碍

真正让我坐直的，是它写无障碍那部分。

你猜大多数人怎么写「要注意无障碍」？就这么写，「要注意无障碍」，一句正确的废话，模型看了等于没看。这个仓库没这么糊弄。它把无障碍拆成一串能执行的硬动作：颜色对比要过 WCAG，标签得用语义化 HTML，键盘能不能 Tab 到每个控件，聚焦的时候有没有那圈高亮，用户要是关了动效你的动画收不收。

你看出门道没有。这些本来都是软要求，是那种「最好做一下」的建议，它全给你抠成了能逐条检查的硬条目。一套系统提示词真正的分量，不在它写了多少条规则，在它把每一条软要求都压成了模型能照着做的硬动作。我那本规则书前几十条也全是空泛的「要口语化」「别太 AI」，后来发现模型根本接不住，才慢慢改成「单句超过二十个汉字就拆开」「破折号一个都不许出现」这种能拿 grep 扫的死规矩。这个仓库等于把这条弯路，在设计领域又走了一遍，还走得比我彻底。

它把审查做成了技能

十四个技能里，有五个是专门做审查的，我看到就笑了。

有一个叫 ai-slop-check，直译就是「查一查这稿子有没有 AI 垃圾感」。还有查无障碍的、查交互状态全不全的、查视觉节奏的。这不就是我干的活吗。

我那套流水线里专门有一个环节，出完稿再回头查一遍 AI 味，逐段打分，哪段读着像机器写的就打回去重写。我甚至有个 skill 专门把每次查出来的毛病，自动提炼成新规则塞回那本规则书，越用越严。区别是我查文字，它查设计，那个动作却一模一样。好东西很少是一次写成的，多半是写完再拿一套审查规则筛出来的。一个成熟的提示词系统，光会生产远远不够，它得自带一套挑自己毛病的机制。这一点，这个仓库比很多人想得深。

触发器我踩过同款坑

有个细节我看到差点从椅子上弹起来。

它专门写了一条：这些前沿模型对可选能力默认是够不着的，你不主动写清楚什么时候该调它，它就想不起来用，所以每个技能都得配一个明确的触发器。你知道我什么反应吗。我在自己那套 skill 里，几乎一模一样的话早就写下来了，我写的是「不要 undertrigger」，翻成人话就是别让它该触发的时候没触发。

我那个拉资讯、做选题的 skill，description 我压根没拿来描述它是干嘛的，我拿它写「什么情况下你就该来叫我」，写成一个开关。当时是被坑出来的。我在对话里丢一句「看看今天有什么值得写的」，它干等着不动，非得我把它名字点出来才肯接活。你以为已经写清楚了，它却理解成这活可有可无，就是这种落差逼着我回头把那行 description 改硬：符合这几个条件，你别问，直接跳出来。这个逆向出来的仓库，把这条坑总结成了一句设计原则。同一个坑，一个写设计的团队，和我这个写内容的，各自摔进去又各自爬出来，爬出来的姿势还一样。

放权那条边界怎么划

还有一条我拿不太准，但值得说。

它给模型定了个自主原则：碰上小决定，自己挑个合理的选项、记一笔，别停下来问人。这条我一半点头一半犹豫。点头是因为，让 agent 每走一步都回头问你，那还不如自己上手干。我前几天刚写过一篇，一个自治的模型一晚上把我平时手动盯的一整条运维活跑完了，那种不打断的顺畅确实爽。

犹豫是因为，「小决定」这三个字的边界谁来划。改个按钮颜色算小决定，那删掉一段它觉得多余的代码算不算。我逆向 Claude Code 那两回，最扎心的就是它擅自做了主还不吭声。所以放权这件事，我现在的态度是：能自己定的让它定，但一定得留痕，你事后翻得到它替你做过哪些主，这条底线松不得。

你也该抠一条死规则

读完这份逆向出来的提示词，我最大的收获跟设计知识关系不大。我又一次被提醒：系统提示词已经是一种正经的工程资产了。

它值得你像写代码一样去分章、抠死规则、配审查、划放权边界。你给 AI 写的那本规则书，就是你俩协作的地基，地基含糊，上面盖什么都晃。

给你一个今晚就能做的最小动作。打开你自己那个 CLAUDE.md，或者随便哪个 prompt，找出里面最含糊的那一条，就是那种你自己读着都觉得模型肯定接不住的「要注意」「尽量」「最好」，把它改成一条能检查的硬动作。一条就行，你会立刻感觉到不一样。

我想问问你，你给 AI 写过最长、最较真的一条规则是什么？或者反过来，你被它的自作主张坑过没有，那次它擅自替你做了什么主？评论区聊聊。
