Fable 5 强到一句话自主登月，我却没让它顶我那 90% 的日常

发表于 2026-07-17 更新于 2026-07-21 分类于 观点 阅读次数： 本文字数： 2.6k 阅读时长 ≈ 2 分钟
Claude Fable 5 是 Anthropic 最强通用模型，Cursor 的 CursorBench 以 Max effort 拿下 72.9% 新高，一句提示能自主登月，而 Opus 跑十几个小时没成。但它默认不开、要手动选，默认仍是 Sonnet 5。我按自己的模型分工阶梯判断，它该顶的不在九成日常活，在没人盯、要跑很久的长异步任务。

哈喽，我是飞飞。

先说清楚，这篇不算又一篇「谁又超了谁」的榜单文。我前两天刚写过 Kimi K3 在前端榜压过 Fable 5，那篇讲的是榜单信号还不够我从 Claude Code 搬家。更早那篇「三个模型接力跑 16 小时」，讲的是流水线交接怎么掉链子。这篇换个问题：Fable 5 这个新面孔，我到底该给它派什么活。

我最初的反应，大概跟你一样。Anthropic 说 Fable 5 是它现在最强的通用模型，那我该不该把主力直接换成它？我掂量了半天，发现自己想岔了。它该顶的班，不在我那九成的日常活。

先给我那队模型排了班

你可能没意识到，一个天天用 Claude Code 的人，脑子里其实挂着一张排班表。

我的活是这么分的。简单的编辑、改格式、顺语法，这类杂活我压给 Haiku，它能扛我六七成的日常。真正写代码、改 bug、加功能，主力是 Sonnet，一个人干掉我九成的活。只有整模块重构、一次要动十几个文件那种硬骨头，我才切 Opus。

为什么卡得这么死？因为贵。每个月给 Anthropic 的钱不算少，让 agent 干一件正经活之前，我都会下意识在心里过一遍秤：这活值不值得动用 Opus。就上周，我批量改二十个 frontmatter 字段，手一滑差点点成 Opus，反应过来赶紧切回 Sonnet。这种「用之前先过秤」的绷着感，就是「贵」最真实的样子。

这套分法，是账单和翻车一点点逼出来的，一开始我压根没这么讲究。我光写文章这一条流水线，就挂着十几个 skill：拉资讯的、查重的、配图的、发公众号的。每一个背后调哪个模型，我都得盯着，不然一个 Haiku 能干的活悄悄爬到 Opus 上，一个月的额度就这么漏光。还有一次贪快，让 Sonnet 在同一个会话里连撑六个钟头，它到后面开始重复生成我早写过的函数，还把旧逻辑改错了。从那以后我学乖：活要按大小分，脑子要按贵贱派。所以我对「哪一棒用哪个脑子」有点洁癖。新来一个更强的，我反而更谨慎。

所以我这张排班表，排的从来跟谁最聪明没关系。看的是这活配得上多贵的脑子。新来一个模型，我头一件事不是掂量它多强，而是先看它该站哪个坑。

Fable 5 到底新在哪

那 Fable 5 强在哪？我去翻了 Anthropic 的官方博客，也看了 Cursor 那边的实测。

官方给它的定位很干脆：专门干长时间、多步骤的复杂异步活。说人话就是，那种你交代一句、它自己闷头跑很久、中间不用你盯的任务，比如深度调研、尽职调查、一摞文档挨个过一遍。

真正让我坐直的是 Cursor 的实测。他们的模型评估负责人 Nate Schmidt，在自家的内部基准 CursorBench 上，给 Fable 5 开 Max effort（也就是把推理档位拉到最满）跑出 72.9%，创了新高。

更有画面感的是一个航天模拟器测试。他只给一句话：造一枚火箭，送上月球。Fable 5 没有一头扎进去，它先规划了一套多阶段策略，先发一次轨道试飞收集遥测数据，再去登月，几个小时就搞定了。

同一个测试换成 Opus 呢？它会一次次发射，在轨道上烧光燃料，加更多燃料，结果火箭太重连大气层都冲不出去，来回折腾十几个小时也没成。Schmidt 一句话点破：Opus 做的是局部推理，Fable 做的是全局推理，它脑子里装的是整场任务。

全局推理补的是哪个洞

「全局推理」这四个字，我一开始也当它是官方话术。直到把它接到自己踩过的坑上，才咂摸出味道。

我那条博客发布流水线，是最放手的一档：push 一下，自动构建、部署、推搜索引擎收录，全程没人盯。有一次负责部署的一个令牌悄悄过期了，构建每次都显示成功，只有最后一步认证红了。没人盯着 workflow 状态，镜像站就这么挂了整整一天，我隔天才发现。

这种坑的要害，不在某一步做错了，在没有一个脑子始终记着「整件事到底干成没有」。每一棒单看都对，合起来却掉了链子。

登月那个例子突然就通了。Opus 的局部推理，活脱脱就是我那条流水线的毛病：盯着眼前这一步，燃料够不够、这一棒接没接上，却没人把整场任务扛在脑子里。Fable 5 的全局推理，补的正是这个洞。它适合的，是那种一口气跑很久、中途没人盯、还得始终记着终点在哪的长活。

我后来把这类活想明白了：越是放手、越省事的活，出事你越晚知道。构建那种一步错立刻红脸的活，我盯着就行；可越往上，到那种自己找活、自己收尾、没人实时看着的循环，一旦某一环悄悄断了，它能闷声堆一整天。就像那次令牌过期，构建天天绿着，我却整整晚了一天才发现镜像站已经塌了。这种活最缺的，恰恰是一个从头到尾记着「我到底要干成什么」的脑子。

所以你看，它该顶的班，跟 Sonnet 一点都不打架。Sonnet 干的是我盯着的活，一来一回；Fable 5 该接的，是那种我撒手不管、它自己闷头跑到底的长活。

我不会派给它的活

反过来，有几类活我不会交给它，哪怕它更强。

我九成的日常编码不会给。那些活是我坐在屏幕前，一句一句喂、一步一步看的，要的是快和便宜，不指望它把整场都想通。这块是 Sonnet 的主场，换 Fable 5 又慢又费。

改个变量名、补段注释、跑个一次性脚本，更不会给。这种活我恨不得用最低档位图快，上 Max effort 纯属杀鸡用牛刀，那 72.9% 背后的推理成本，最后都得我替它买单。

还有一层现实得提醒你：Fable 5 在 Cowork 里默认压根不开，你得手动选它，默认仍然是 Sonnet 5。这个默认设置其实挺诚实，Anthropic 自己也把它当成非常规活才叫醒的那一档，不当日常主力使。我这张排班表，跟它这个默认，算是想到一块去了。

今天就能定的一个动作

如果你也天天在几个模型之间切，今天可以做件很小的事。

先别急着问 Fable 5 强不强，翻出你手里那条最长、最没人盯的活。可能是一个跑一整晚的批处理，可能是一条自动发布流水线，也可能是一份要连着读几十个文档的调研。就那条活，给 Fable 5 留一个候选位。

我自己还没把它正式排上班。它默认不开，我也没大规模压过真活，所以这篇给你的是判断，算不上实测报告，这点我得说在前头。但我那把用了大半年的分工尺子已经给出答案：它要顶的不在主力那班，在我流水线上最怕静默出事的那条长活。

绕了一圈我才想明白，新模型来了，该改的往往不在换谁当主力，在给它单开一个班次。

你手里那条最怕没人盯着就掉链子的长活，现在是谁在扛？评论区聊聊，我想看看你的排班表，跟我的差多少。
