AI Agent 会成为新的操作系统吗，我天天用它当入口，但还没准备好删掉那些 App

发表于 2026-06-25 更新于 2026-07-13 分类于 观点 阅读次数： 本文字数： 2.7k 阅读时长 ≈ 2 分钟
纳德拉早前在一档播客里放话应用层会在 agent 时代坍缩，我天天用 Claude Code 当入口驱动十几个 MCP 工具，对这个判断有真实体感，但我认为 App 不会立刻消失，能交给 agent 当后端的是流程类服务，需要你盯着屏幕拿主意的那部分，入口还是落回 App。

哈喽，我是飞飞。

这几天我刷推、刷朋友圈，老看到同一类话在转：微软的纳德拉早前在一档播客里放话，应用层会在 agent 时代整个坍缩，操作系统加 App 那一套要让位给「agent 的时代」了。他还有句更狠的，把那些 SaaS 应用说成「就是套了业务逻辑的增删改查数据库」。一句话，你那些 App 要过时了，以后人跟电脑打交道，主入口是 agent。

我看到这条的第一反应不是震惊。说真的，我反而有点想笑，因为我每天干活，差不多就是这么干的。

我现在的工作台，已经是 agent 当入口了

说个我自己的真实状态。我光「写文章」这一条流水线，就挂着十几个自定义 skill：拉 AI 资讯的、做选题打分的、搜资料的、查重的、配封面图的、把稿子转成公众号格式的、推到微信草稿箱的。这些全长在 Claude Code 上，靠它认 frontmatter、调 hook、按我定的规则一个个跑。

我打开终端，敲一句「帮我把今天这个选题写出来」，后面这一长串活，是 agent 自己串起来调的：调搜索 MCP 去查资料，调画图 MCP 去出封面，调浏览器去抓网页，调本地脚本去渲染。我不再一个个去开那些独立的网页、独立的工具、独立的后台。我面对的就一个入口，就是那个对话框。

这套东西养着也烧钱。每个月给 Anthropic 那笔订阅不算少，让 agent 动一件正经活之前，我都会下意识掂量一下值不值得动用 Opus，杂活就手动切个便宜模型省点额度。一个你得算着账才舍得用的东西，它在你工作流里早就脱离了玩具的范畴，成了每天离不开的那块地基。

传统那条路是「用户 → App → 服务」，你想搜资料开浏览器，想画图开画图网站，想发文章登公众号后台，每个 App 是一道门，你得自己挨个推开。我现在这条是「用户 → agent → 一堆服务」，agent 站在前面当总机，背后那些工具退成了它随手调的零件。纳德拉说的那个「应用层坍缩进 agent」，搁我这儿不是预言，是已经发生的日常。

所以纳德拉这半句，我先认下来。这事是真的，方向也是真的。

把这条路撑起来的，是一个叫 MCP 的协议

这里得插一句技术背景，不然「agent 怎么就能调动一堆工具」这事说不清。

让 agent 能挨个调起外部工具的，是个叫 MCP 的东西，Model Context Protocol，Anthropic 在 2024 年 11 月放出来的开放标准。它解决的是一个很笨重的老问题：以前你要让 N 个 AI 应用接上 M 个工具，得一对一写连接器，10 个应用接 100 个工具就是上千个对接，谁也扛不住。MCP 把这层标准化了，工具按一套统一的格式描述自己，agent 照着这格式就能调，再不用一家一家单独适配。

这玩意儿火得不正常。按维基百科和几家技术媒体的梳理，OpenAI 在 2025 年 3 月官方接了 MCP，把它铺进 Agents SDK 和 ChatGPT；谷歌的 Gemini 跟着支持；到 2025 年 4 月，社区里的 MCP server 下载量已经过了 800 万。年底 12 月，Anthropic 干脆把 MCP 捐给了 Linux 基金会下面新成立的 Agentic AI Foundation，连 OpenAI 都进去当了联合发起方。竞争对手不会平白无故去用对家的协议，除非这个网络效应已经大到拦不住了。

为什么我要讲这一段？因为正是这层协议，让「agent 当入口、App 退成后端零件」从我一个人的玩法，变成了整个行业都在往里砸钱的方向。我那十几个 skill 调工具，底下走的就是这套口子。

但要我现在删掉手机里的 App，我做不到

讲完我认同的那一半，得把另一半也摊开。纳德拉是微软的 CEO，他这话是带着公司赌注说的，是一个判断、一个押注，不是已经验证过的事实。换我这个天天用 agent 的人来看，App 短时间内不会消失，而且有一大类活，它就该留在 App 里。

我自己就有体感。前两天我博客的发布流水线静默挂了一整天。那条流水线全是自动化的，我 push 代码上去，它自己构建、自己部署、自己把新链接推给搜索引擎收录。结果一个 GitHub 的令牌过期了，最后一步推送认证失败，红了一整天。构建明明是绿的，就最后那一下挂了，没人盯着，我是隔了一天主动去翻 workflow 状态才发现的。更要命的是主站靠另一条独立的 Docker 流水线还活着，那个镜像站却就这么静默挂了一天，没人知道。

这件事给我提了个醒：把流程整条交给 agent 自动跑，最贵的隐藏成本不在算力。它藏在没人盯着的时候，失败会悄无声息地堆起来。需要你眼睛盯着、需要你当场拿主意的那部分，恰恰是 agent 替不掉的。

界面这条也一样。我查到一份做交互设计的分析（markswebb，2026 年），里头说得挺实在：很多公司图新鲜，想把所有操作都塞进对话框，搞「聊天优先」，到了规模上反而崩。用户被迫在聊天和按钮之间来回切，同一个操作在两边规则还不一样，越用越乱。你点一个按钮就办完的事，非要打一段话描述清楚，这哪是进步，分明是倒退。

还有个常被吹的能力，让 agent 直接操作你的电脑屏幕、自己点按钮。这个我得说句冷的：OpenAI 的 Operator、Anthropic 的 Computer Use，到现在都还是研究预览或者 beta，没给大多数用户默认开。能不能稳定地替你点完一连串操作，业界自己心里也没底。

所以我的分法：能流程化的交出去，要拿主意的留下来

绕了一圈，我自己心里那条线慢慢清楚了。

agent 适合接管的，是流程类的服务。目标明确、步骤可拆、跑错了能回滚的那种活，交给它当后端，你只管在入口下一句指令。我那十几个 skill 干的就是这类事，搜、查、转格式、推草稿，链条长但每一步都有标准答案。这部分我巴不得 agent 全包了，省得我一个个开窗口。

留在 App 里的，是要你盯着屏幕、当场拍板的那部分。你刷银行账单核对每一笔、你在地图上比划着选路线、你翻照片一张张挑、你看那个红了一整天的流水线状态。这些活的价值就在那个视觉界面本身，在你眼睛一扫就抓到异常的那种直觉上。把它硬塞进对话框，是把好用的东西做难用了。

说到底，App 不会被 agent 一锅端，更像是被它分了层。前台那个总入口，越来越大概率是个 agent；后台那些干脏活累活的服务，继续以 App 或者 API 的形态活着，只是你越来越少直接去碰它们。纳德拉说的「坍缩」，坍的是入口，不是整栋楼。

至于 agent 会不会成为新的操作系统，我的态度是：它正在长成入口层，这个我信，我每天都在用。但操作系统底下那一整套调度、权限、可靠性的地基，现在的 agent 还远没接住。我那条挂了一整天的流水线，就是这地基还没夯实的活证据。它在往那个方向走，但还没到。

那么问题来了：你手机里哪个 App，是你最愿意直接交给 agent 替你打理、再也不想自己点开的？又是哪一个，你打死也得自己开着、亲眼盯着才放心？
